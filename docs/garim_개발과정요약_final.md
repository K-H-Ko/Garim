# 🚀 GARIM 기술 개발 과정 및 문제 해결 요약

이 문서는 프로젝트 진행 과정(v1~v20)에서 발생한 주요 기술적 한계를 극복하고, 플랫폼의 핵심 기능을 고도화하며 겪은 **문제 해결 과정**을 요약한 리포트입니다.

---

## 1. AI 모델 선정 및 교체 과정 (성능 및 정확도 최적화)
- **문자 인식(OCR) 모델 변경: `Naver Clova API` ➔ `PaddleOCR (PP-OCRv5)`**
  - **문제 상황**: 초기에는 한글 인식률이 가장 높은 외부 API(Naver Clova)를 사용했으나, 영상 프레임마다 반복 호출 시 막대한 비용이 발생하고 통신 지연(타임아웃)이 컸습니다.
  - **해결 과정**: 로컬 환경에서도 동작하며 한국어 인식에 특화된 `PaddleOCR` 경량 모델(PP-OCRv5)로 전격 교체했습니다. 외부 의존성을 없애 100% 무료 오프라인 파이프라인을 구축했고, 여기에 자체 조건부 고배율(2.0x) 재탐지 로직을 추가해 네이버 수준의 탐지율을 확보했습니다.
- **배경 복원(Inpainting) 모델 변경: `Stable Diffusion` ➔ `LaMa + Telea 혼합`**
  - **문제 상황**: 처음엔 가장 완벽한 배경 복원을 위해 생성형 AI인 `Stable Diffusion` 인페인팅을 도입했습니다. 하지만 개인정보를 지운 자리에 모델이 '가짜 글자'나 엉뚱한 이미지를 임의로 그려 넣는 치명적인 환각(Hallucination) 부작용이 발생했습니다.
  - **해결 과정**: 새로운 텍스트 생성이 배제된, 순수 픽셀 복원 특화 모델인 `LaMa`와 보간 알고리즘(`Telea`)을 결합한 **적응형 인페인팅**으로 방향을 전환했습니다. 배경의 엣지 밀도를 분석해 복잡도에 따라 두 알고리즘을 자동 분기하여 엉뚱한 글자 생성 없이 이질감을 완벽히 제거했습니다.
- **음성 인식(STT) 모델 변경: `기본 Whisper` ➔ `Faster-Whisper`**
  - **문제 상황**: 초기 OpenAI의 기본 `Whisper` 모델은 연산량이 너무 많아 GPU 메모리 소모가 극심했고, 영상 처리 시간이 실제 영상 길이보다 길어지는 심각한 지연 현상이 있었습니다.
  - **해결 과정**: 엔진 최적화가 적용된 `Faster-Whisper` 모델로 교체하여, 한글 인식 정확도는 완벽히 유지하면서 처리 속도를 3배 이상 끌어올리고 메모리 점유율을 대폭 낮췄습니다.

## 2. 하이브리드 자동화 워커 구조 구축 (메모리 부족 및 병목 현상 해소)
- **문제 상황**: 초기 단일 서버 구조에서 영상 처리, Whisper(음성 인식), EasyOCR(문자 인식), 인페인팅(Inpainting) 등의 무거운 딥러닝 모델을 모두 한 곳에 올려서 실행하다 보니, 심각한 메모리 부족 현상과 응답 지연(타임아웃) 에러가 지속적으로 발생했습니다.
- **해결 과정**: 
  - 전체 분석 과정을 5단계로 나누어 분산 환경으로 재설계했습니다. 가벼운 CPU 연산(문자 인식, 데이터 병합, 화면 출력 준비)은 **로컬 워커**로 내리고, 무거운 GPU 연산(음성 인식, 마스킹)은 외부 **Colab 워커**로 위임하는 전 구간(End-to-End) 하이브리드 연동망을 구축했습니다.
  - 워커 간 통신을 위해 데이터베이스 잠금 현상을 유발하는 쿼리 통신 대신, Cloudflare 터널을 결합한 `result.json` 결과물 파일 공유 방식으로 변경하여 완벽한 무인 자동화와 성능 최적화를 달성했습니다.

## 3. 마스킹 품질 고도화: 가짜 합성 폐기 및 원본 100% 보존
- **문제 상황**: 초기엔 탐지된 개인정보 자리에 Gemini LLM으로 가짜 데이터를 만들고 3D 왜곡 변환(`warpPerspective`)으로 폰트를 합성하려 했으나, 이질감이 커서 서비스 품질이 저하되었습니다. 또한 마스킹 영역이 확장되면서 표의 구분선이나 바로 옆 글자까지 깎여나가는 훼손 현상이 발생했습니다.
- **해결 과정**:
  - **합성 전면 폐기 및 비가역 삭제 집중**: 가짜 데이터 합성 노선을 과감히 폐기하고, 어떠한 복원 툴로도 살릴 수 없도록 픽셀을 완전히 삭제하는 동시에 배경만 감쪽같이 되살리는(LaMa+Telea) 방향으로 아키텍처를 전면 수정했습니다.
  - **표 격자선 및 인접 글자 2중 보호막**: 형태학적 연산(Morphology)으로 표의 가로/세로 격자선 및 경계 그림자를 실시간으로 감지해 지우기 대상에서 제외했습니다. 또한, 지우지 말아야 할 주변 원본 글자에는 **사방 3px의 안심 마진(보호막)**을 수학적으로 차단 적용하여 인접 데이터 훼손 0%를 달성했습니다.
  - **음성(묵음 처리) 최적화**: 시각적 마스킹에 맞춰 FFmpeg 기반으로 원본 오디오 파형을 정밀한 밀리초 단위로 분리하고 무음 트랙과 재합성하여 오디오 밀림 현상을 완벽히 제어했습니다.

## 4. 문자 탐지 정밀화 및 영상 픽셀 객체 추적(NCC) 도입
- **문제 상황**: 동영상 처리 시 영상이 흔들리거나 글씨가 빠르게 지나가면 프레임마다 개인정보를 완벽히 탐지(OCR)하기 어려워 마스킹이 깜빡이거나 누락되는 구간이 발생했습니다.
- **해결 과정**:
  - **픽셀 기반 양방향 추적(NCC)**: 단순히 탐지된 시간만 지우는 것이 아니라, 템플릿 매칭 기반의 픽셀 추적 알고리즘을 도입해 탐지된 글자 픽셀이 이동하고 회전(Affine 변환)하는 것을 끝까지 쫓아가며 마스킹하도록 고도화했습니다.
  - **과잉 마스킹 및 오탐 방지**: 화면에서 카드가 사라지거나 다른 장면으로 넘어갈 경우, 픽셀 일치율(NCC 임계치 0.5)을 실시간 검증하여 스스로 마스킹 추적을 즉시 중단하게 만들어 빈 화면에 검은 박스가 떠다니는 현상을 원천 차단했습니다.
  - **파편화된 탐지 박스 병합**: 탐지된 한글 주소가 파편화되어 쪼개지는 현상을 막기 위해 탐지 박스를 하나로 합치는(Merge) 알고리즘을 최적화하여 하나의 단일 개인정보 블록으로 정교하게 관리했습니다.

## 5. 결제 및 구독 시스템 보안 강화와 재시도 대기열 도입
- **문제 상황**: 30일 주기의 정기 구독 결제를 도입하면서, 유저의 자동 결제 키를 데이터베이스에 그대로 저장하면 보안에 취약해지는 문제가 있었습니다. 또한 잔액 부족 등으로 결제 실패 시 예외 처리가 부족했습니다.
- **해결 과정**:
  - 토스 페이먼츠 결제 API를 연동하고, 결제 키는 데이터베이스 내장 함수(`pgp_sym_encrypt`)를 활용해 데이터베이스단에서 강력하게 암호화하여 저장했습니다.
  - 한도 초과 등으로 자동 결제가 실패할 경우를 대비해 **재시도 대기열 스케줄러**를 구현하고, 최종 실패 시 자동으로 기본 요금제로 강등 처리되는 무인 결제 주기를 완성했습니다.
