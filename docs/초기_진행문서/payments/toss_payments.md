# Toss Payments 결제 연동 시 기술 고려사항

토스페이먼츠(Toss Payments) 결제 연동 작업을 설계하고 구현할 때 시스템의 보안성, 무결성, 안정성을 확보하기 위해 다음 5가지 기술적 고려사항을 준수해야 합니다.

---

## 1. 사전 주문 생성 및 결제 금액 검증 (위변조 방지)

- **사전 주문 등록 (Pre-registration)**: 프론트엔드에서 Toss 결제창을 호출하기 전에 반드시 백엔드 API를 호출하여 임시 주문 또는 결제 요청 레코드를 데이터베이스에 생성해야 합니다. 이 단계에서 고유한 주문 ID(`orderId`)와 결제 금액(`amount`)이 임시 결제 상태와 함께 DB에 영속화되어야 합니다.
- **최종 검증 (Double-Check)**: 결제창 완료 후 Toss 측에서 프론트엔드로 리다이렉트되어 전달된 `amount` 및 `orderId` 값을 기반으로 백엔드 승인(Confirm) API를 호출하게 됩니다. 백엔드는 토스페이먼츠 승인 API를 호출하기 전에, **인바운드로 넘어온 결제 완료 금액이 사전에 데이터베이스에 등록해 둔 결제 요청 금액과 완벽히 일치하는지 반드시 검증**해야 합니다. (클라이언트 단의 자바스크립트 값 변조 및 결제 금액 조작 방지)

## 2. API 요청 멱등성(Idempotency) 처리

- **중복 승인 차단**: 사용자가 리다이렉트 페이지에서 새로고침을 누르거나 네트워크 지연 등으로 인해 동일한 `orderId`에 대해 백엔드 승인 API가 중복 호출될 수 있습니다.
- **사전 상태 검사**: 백엔드 승인 API 내부에서 토스페이먼츠 승인 API를 실제로 호출하기 전에, **해당 주문 ID의 결제가 이미 성공 처리(`payments.status = 'success'`)되었는지 데이터베이스를 먼저 확인**해야 합니다. 이미 처리된 건이라면 추가적인 Toss API 호출 없이 성공 결과를 즉시 반환하도록 하여 중복 과금 및 네트워크 에러를 방어해야 합니다.

## 3. 데이터베이스 트랜잭션과 일관성 유지 (Atomicity)

토스페이먼츠 승인 API 호출이 성공적으로 완료(`200 OK`)된 직후, 백엔드 내부 데이터베이스의 일관성을 맞추기 위해 다음 작업들을 **단일 트랜잭션**으로 처리하고, 하나라도 실패 시 전체 롤백(Rollback)을 수행해야 합니다.

- **결제 이력 기록**: `payments` 테이블에 상태(`success`), 결제 정보(`paid_at`), Toss 측 트랜잭션 키(`pg_transaction_id`에 Toss `paymentKey` 저장) 반영
- **구독 상태 설정**: `subscriptions` 테이블에 유저의 구독/이용권 시작일(`started_at`), 종료일(`ended_at`), 구독 상태(`status = 'active'`) 최신화
- **쿼터/크레딧 지급**: 이번에 갱신한 `plans` 테이블(구독) 또는 `credit_plans` 테이블(크레딧 충전)의 기준에 따라 유저의 잔액(`user_credit_balances.balance`)을 충전 및 지급하고, 내역을 `credit_ledger`에 기록

## 4. 결제 유실 방지를 위한 웹훅(Webhook) 구성

- **이탈 상황 대응**: 사용자가 결제 승인 완료를 처리하기 전 브라우저 탭을 닫거나 모바일 환경에서 이탈한 경우, 혹은 가상계좌(무통장 입금) 결제를 진행하는 경우 프론트엔드 화면 리다이렉트(successUrl)를 통한 결제 승인 흐름이 유실될 수 있습니다.
- **웹훅 연동**: 토스페이먼츠 관리자 콘솔에 백엔드 웹훅 엔드포인트(예: `/api/payment/webhook`)를 구성하여, 결제 성공 또는 가상계좌 입금 완료 시 Toss 서버가 백엔드로 상태 변경 신호를 직접 전송하도록 해야 합니다. 웹훅 수신 시에도 승인 프로세스와 동일한 금액 검증 및 트랜잭션 처리(멱등성 검사 포함)가 수행되어야 합니다.

## 5. 취소 및 환불 프로세스 연동

- **취소 이력 업데이트**: 사용자가 결제 취소 또는 환불을 요청하는 경우 백엔드는 토스페이먼츠 환불 API를 호출해야 합니다. 승인 성공을 수신하면 `payments.status`를 `cancelled`로 업데이트하고 `refunded_at`을 기록합니다.
- **구독 및 한도 회수**: 환불 승인 즉시 해당 결제 건과 묶여 있는 구독 정보(`subscriptions`)의 상태를 비활성화(`expired` 또는 `cancelled`)하고, 유저에게 부여했던 잔여 쿼터/크레딧(`user_credit_balances.balance`)을 회수하는 환불 정책 비즈니스 로직을 적용해야 합니다.
