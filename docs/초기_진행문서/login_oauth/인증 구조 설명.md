# 인증 구조 설명

이 문서는 현재 `Human_Final_PJ` 프로젝트에 구현된 인증 구조를 기준으로 정리한 문서입니다.

현재 인증은 **OAuth-only** 방식입니다. 이메일/비밀번호 기반 로그인, 회원가입, 비밀번호 변경, 로컬 패스워드 인증은 사용하지 않습니다.

## 1. 전체 구조

```text
Frontend
  -> OAuth 로그인 버튼 클릭
  -> Backend OAuth start API 호출
  -> OAuth Provider 인증
  -> Backend OAuth callback
  -> users 조회/생성 및 상태 확인
  -> 자체 Access JWT / Refresh JWT 발급
  -> Redis에 session_id 기준 세션 저장
  -> JWT를 HttpOnly Cookie로 저장
  -> Frontend는 /auth/me 또는 /auth/status로 로그인 상태 확인
```

주요 구성은 다음과 같습니다.

| 구성            | 역할                                                                  |
| --------------- | --------------------------------------------------------------------- |
| OAuth Provider  | Google, Kakao, Naver 등 외부 로그인 제공자                            |
| Backend         | OAuth callback 처리, 사용자 조회/생성, 자체 JWT 발급, Redis 세션 관리 |
| Access JWT      | 보호 API 요청 인증용 짧은 수명의 JWT                                  |
| Refresh JWT     | Access JWT 재발급용 긴 수명의 JWT                                     |
| HttpOnly Cookie | 브라우저에 JWT를 저장하되 JavaScript 직접 접근을 막음                 |
| Redis Session   | `session_id` 기준 로그인 세션과 refresh 검증 정보를 저장              |
| Frontend        | OAuth 시작, 인증 상태 조회, 로그아웃 요청만 수행                      |

## 2. 지원 OAuth Provider

현재 `backend/services/oauth.py` 기준 provider는 다음과 같습니다.

| Provider | 환경 변수                                                               |
| -------- | ----------------------------------------------------------------------- |
| Kakao    | `KAKAO_CLIENT_ID`, `KAKAO_CLIENT_SECRET`, `KAKAO_REDIRECT_URI`          |
| Google   | `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `GOOGLE_REDIRECT_URI`       |
| Naver    | `NAVER_CLIENT_ID`, `NAVER_CLIENT_SECRET`, `NAVER_REDIRECT_URI`          |
| Facebook | `FACEBOOK_CLIENT_ID`, `FACEBOOK_CLIENT_SECRET`, `FACEBOOK_REDIRECT_URI` |
| X        | `X_CLIENT_ID`, `X_CLIENT_SECRET`, `X_REDIRECT_URI`                      |

프론트 로그인 화면은 현재 Kakao, Naver, Google 버튼을 사용합니다.

## 3. API 경로

라우트는 `backend/routes/oauth.py` 기준입니다.

| Method | URL                                  | 설명                         |
| ------ | ------------------------------------ | ---------------------------- |
| GET    | `/auth/{provider}/start`             | OAuth 로그인 시작            |
| GET    | `/auth/{provider}/callback`          | OAuth callback               |
| GET    | `/auth/oauth/{provider}`             | OAuth 로그인 시작 호환 경로  |
| GET    | `/auth/oauth/{provider}/callback`    | OAuth callback 호환 경로     |
| GET    | `/auth/me`                           | 현재 로그인 사용자 조회      |
| GET    | `/auth/status`                       | 로그인 상태 조회             |
| POST   | `/auth/refresh`                      | Access/Refresh JWT 재발급    |
| POST   | `/auth/logout`                       | 현재 세션 로그아웃           |
| DELETE | `/auth/me`                           | 회원 탈퇴                    |
| DELETE | `/auth/sessions/{session_id}`        | 특정 세션 삭제               |
| DELETE | `/auth/sessions`                     | 현재 사용자의 모든 세션 삭제 |
| PATCH  | `/auth/admin/users/{user_id}/status` | 사용자 상태 변경             |

프론트 `frontend/src/utils/api.js`는 기본 API 주소를 `http://127.0.0.1:8000`으로 사용하며, `VITE_API_BASE_URL`로 덮어쓸 수 있습니다.

## 3.1 OAuth 계정 식별 기준

OAuth Provider가 내려주는 이메일은 로그인 식별자가 아니라 provider 계정의 이메일 속성입니다.

예를 들어 Naver OAuth 계정의 이메일이 Google 또는 Daum 메일 주소일 수 있으므로 `users.email`만으로 사용자를 고유 식별하면 안 됩니다.

현재 구현 기준은 다음과 같습니다.

| 데이터 | 용도 |
|---|---|
| `users.email` | 연락용/표시용 이메일. 단독 unique 식별자로 사용하지 않음 |
| `oauth_accounts.provider` | OAuth 제공자 |
| `oauth_accounts.provider_email` | OAuth 제공자가 내려준 이메일 |
| `oauth_accounts.provider_user_id` | OAuth 제공자가 내려준 사용자 고유 ID |

로그인 시 기존 계정 조회는 다음 기준으로 수행합니다.

```text
oauth_accounts.provider = provider
AND (
  provider_email이 있으면 lower(oauth_accounts.provider_email) = lower(provider_email)
  OR oauth_accounts.provider_user_id = provider_user_id
)
```

DB 인덱스 기준:

```sql
DROP INDEX IF EXISTS idx_users_email_unique;
CREATE INDEX IF NOT EXISTS idx_users_email ON users (email);
CREATE UNIQUE INDEX IF NOT EXISTS idx_oauth_accounts_provider_email_unique
ON oauth_accounts (provider, LOWER(provider_email))
WHERE provider_email IS NOT NULL;
CREATE UNIQUE INDEX IF NOT EXISTS idx_oauth_accounts_provider_user_unique
ON oauth_accounts (provider, provider_user_id);
```

즉 같은 이메일이라도 provider가 다르면 서로 다른 OAuth 계정으로 저장할 수 있습니다.

기존 DB에 이미 `idx_users_email_unique`가 있다면 위 인덱스 변경 SQL을 한 번 반영해야 합니다.

## 4. 로그인 흐름

```text
1. 사용자가 프론트에서 OAuth 버튼 클릭
2. 프론트가 /auth/{provider}/start 로 이동
3. 백엔드가 OAuth Provider 인증 URL로 redirect
4. Provider가 /auth/{provider}/callback 으로 code/state 전달
5. 백엔드가 code로 Provider access token 요청
6. Provider userinfo 조회
7. oauth_accounts 테이블에서 provider + provider_email 우선, provider + provider_user_id fallback 기준 조회
8. 없으면 OAuth 사용자 생성
9. users.status가 ACTIVE인지 확인
10. Access JWT / Refresh JWT 발급
11. Redis에 auth:session:{session_id} 저장
12. access_token / refresh_token HttpOnly Cookie 설정
13. 프론트 /dashboard 로 redirect
```

`users.status`가 `ACTIVE`가 아니면 로그인은 실패합니다.

## 5. JWT 구조

### Access JWT

Access JWT는 보호 API 요청 인증용입니다.

기본 만료 시간은 `ACCESS_TOKEN_EXPIRE_SECONDS`이며, 기본값은 900초입니다.

예시 payload:

```json
{
  "sub": "1",
  "sid": "session_id",
  "jti": "access_token_unique_id",
  "role": "USER",
  "type": "access",
  "exp": 1710000000
}
```

### Refresh JWT

Refresh JWT는 Access JWT 재발급용입니다.

기본 만료 시간은 `REFRESH_TOKEN_EXPIRE_SECONDS`이며, 기본값은 604800초입니다.

예시 payload:

```json
{
  "sub": "1",
  "sid": "session_id",
  "jti": "refresh_token_unique_id",
  "type": "refresh",
  "exp": 1710600000
}
```

## 6. Cookie 정책

JWT는 HttpOnly Cookie에 저장합니다.

```http
Set-Cookie: access_token=...; HttpOnly; SameSite=Lax; Path=/; Max-Age=900
Set-Cookie: refresh_token=...; HttpOnly; SameSite=Lax; Path=/auth/refresh; Max-Age=604800
```

| 쿠키            | Path            | 용도          |
| --------------- | --------------- | ------------- |
| `access_token`  | `/`             | 보호 API 인증 |
| `refresh_token` | `/auth/refresh` | 토큰 재발급   |

`COOKIE_SECURE=true`이면 Secure 쿠키로 발급됩니다. 로컬 HTTP 개발 환경에서는 보통 `COOKIE_SECURE=false`를 사용합니다.

주의할 점:

- HttpOnly 쿠키는 프론트 JavaScript에서 직접 삭제할 수 없습니다.
- 반드시 백엔드가 `Set-Cookie` 만료 응답을 내려줘야 브라우저 Application 탭에서도 삭제됩니다.
- 쿠키 삭제는 발급 당시의 Domain/Path와 일치해야 합니다.
- `localhost`와 `127.0.0.1`은 브라우저 쿠키 기준으로 다른 도메인입니다.

## 7. Redis 세션 구조

Redis에는 `session_id` 기준으로 세션을 저장합니다.

```text
key: auth:session:{session_id}
```

예시 value:

```json
{
  "user_id": "1",
  "refresh_jti": "uuid",
  "refresh_hash": "hashed_refresh_token",
  "provider": "google",
  "role": "USER",
  "user_agent": "...",
  "ip": "...",
  "created_at": "...",
  "expires_at": "..."
}
```

Redis session TTL은 `REDIS_SESSION_TTL_SECONDS`로 설정하며, 기본값은 604800초입니다.

## 8. 보호 API 인증

Access Token 검증은 JWT 검증만으로 끝나지 않습니다.

```text
1. access_token Cookie 추출
2. JWT 서명, exp, type=access 검증
3. payload에서 sid 추출
4. Redis에서 auth:session:{sid} 조회
5. Redis session이 없으면 401
6. session.user_id와 JWT sub 일치 확인
7. users.status 확인
8. ACTIVE가 아니면 401
9. 통과 시 보호 API 실행
```

따라서 Redis session을 삭제하면 Access JWT가 아직 만료되지 않았더라도 보호 API는 401을 반환합니다.

## 9. Refresh 흐름

```text
1. 프론트가 POST /auth/refresh 호출
2. 브라우저가 refresh_token Cookie 전송
3. 백엔드가 Refresh JWT 서명, exp, type=refresh 검증
4. sid 기준 Redis session 조회
5. Redis session이 없으면 401
6. refresh_jti 일치 확인
7. refresh_hash 일치 확인
8. users.status ACTIVE 확인
9. 새 Access JWT / Refresh JWT 발급
10. Redis의 refresh_jti / refresh_hash 갱신
11. 새 Cookie 설정
```

기존 Refresh Token을 다시 사용하면 `refresh_jti` 또는 `refresh_hash`가 맞지 않아 해당 세션을 삭제하고 401을 반환합니다.

## 10. 로그아웃 흐름

프론트 설정 화면의 로그아웃 버튼은 `POST /auth/logout`을 호출합니다. 성공하면 프론트는 `/`로 이동합니다.

백엔드 로그아웃 흐름은 다음과 같습니다.

```text
1. Cookie에서 access_token, refresh_token, garim_auth 추출
2. 토큰 중 하나에서 sid 추출
3. Redis auth:session:{sid} 삭제
4. access_token Cookie 삭제
5. refresh_token Cookie 삭제
6. garim_auth 레거시 Cookie 삭제
7. authenticated=false 응답
```

현재 삭제 응답은 다음 쿠키를 만료시킵니다.

```http
Set-Cookie: access_token=; HttpOnly; SameSite=Lax; Path=/; Max-Age=0
Set-Cookie: refresh_token=; HttpOnly; SameSite=Lax; Path=/auth/refresh; Max-Age=0
Set-Cookie: refresh_token=; HttpOnly; SameSite=Lax; Path=/; Max-Age=0
Set-Cookie: garim_auth=; HttpOnly; SameSite=Lax; Path=/; Max-Age=0
```

`garim_auth`는 현재 OAuth-only 구조의 주 쿠키는 아니지만, 이전 인증 구현 또는 테스트 과정에서 남을 수 있는 레거시 쿠키입니다. 로그아웃 시 함께 삭제하도록 처리합니다.

## 11. 회원 탈퇴

회원 탈퇴는 `DELETE /auth/me`로 처리합니다.

```text
1. access_token 인증
2. users.status = DELETED 처리
3. 해당 user_id의 모든 Redis session 삭제
4. access_token / refresh_token / garim_auth Cookie 삭제
```

탈퇴 이후 `users.status`가 ACTIVE가 아니므로 로그인, refresh, 보호 API 요청은 모두 차단됩니다.

## 12. 계정 정지

관리자 계정 정지는 `PATCH /auth/admin/users/{user_id}/status`로 처리합니다.

```text
1. users.status = SUSPENDED 처리
2. 해당 user_id의 모든 Redis session 삭제
3. 기존 Access JWT는 Redis session을 찾지 못해 401 처리
4. refresh도 Redis session이 없으므로 401 처리
```

## 13. Frontend 동작

프론트는 JWT를 직접 읽거나 저장하지 않습니다.

현재 프론트 인증 관련 API는 `frontend/src/utils/api.js`에 있습니다.

| 함수                         | 동작                              |
| ---------------------------- | --------------------------------- |
| `getOAuthStartUrl(provider)` | `/auth/{provider}/start` URL 생성 |
| `getAuthStatus()`            | `/auth/status` 호출               |
| `getCurrentUser()`           | `/auth/me` 호출                   |
| `refreshAuthSession()`       | `/auth/refresh` 호출              |
| `logout()`                   | `/auth/logout` 호출               |

설정 화면의 로그아웃 버튼은 `logout()` 호출 후 `/`로 이동합니다.

## 14. 실제 구현과 문서 차이 요약

이번 정리에서 기존 문서와 달라진 주요 부분은 다음과 같습니다.

| 항목             | 기존 문서                 | 현재 구현                                                         |
| ---------------- | ------------------------- | ----------------------------------------------------------------- |
| OAuth 시작 경로  | `/auth/oauth/google` 중심 | `/auth/{provider}/start` 사용, `/auth/oauth/{provider}` 호환 유지 |
| Provider         | Google/Kakao 중심         | Kakao, Google, Naver, Facebook, X 설정 존재                       |
| 로그아웃 후 이동 | 명시 부족                 | 프론트에서 `/`로 이동                                             |
| 쿠키 삭제        | access/refresh 중심       | access, refresh(`/auth/refresh`), refresh(`/`), `garim_auth` 삭제 |
| 레거시 쿠키      | 설명 없음                 | `garim_auth` 삭제 및 sid 추출 fallback 처리                       |
| OAuth 계정 식별 | 이메일 설명 부족          | `users.email` 단독 unique 제거, `oauth_accounts(provider, provider_email)` 복합 unique 추가 |
| DB 초기화 SQL    | `idx_users_email_unique` 기준 | `idx_users_email`, `idx_oauth_accounts_provider_email_unique` 기준 |
| DB 설계서        | 기존 Excel 기준           | `Garim_DB_Design_final_clean_v2.xlsx`에 현재 인증/인덱스 구조 반영 |
| Access 검증      | Redis session 확인 원칙   | 실제 코드에서도 `sid`로 Redis session 필수 확인                   |
| Refresh 검증     | jti/hash 검증 원칙        | 실제 코드에서도 `refresh_jti`, `refresh_hash` 검증                |
