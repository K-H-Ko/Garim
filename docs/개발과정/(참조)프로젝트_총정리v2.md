# GARIM 프로젝트 비즈니스 기획 및 기술 아키텍처 총정리 v2

> **v2 갱신 핵심**: v1의 "기획 배경·타겟·핵심 가치"는 유지하되, **처리 아키텍처를 신코드 기준으로
> 교체**했다. v1은 `colab_pipeline_report.py`/`colab_pipeline_mask.py` 이원화 구조였으나, 실제로는
> **로컬 CPU 3모듈 + Colab GPU 2모듈로 5분할**되고 **`result.json` 단일 마스터 데이터**를 중심으로
> "탐지→상세보기→선택→선택분만 마스킹"이 흐른다. 또한 **이 파이프라인을 기존 프론트/백엔드 구조에
> 맞춰 연결**하는 원칙을 추가했다.

---

## 1. 서비스 기획 배경 및 비즈니스 목적 (유지)

### 1.1 기획 배경: '퍼즐 도싱(Doxxing)' 차단
개인정보 유출은 DB 해킹뿐 아니라, SNS/브이로그에 무심코 노출된 영상 속 **택배 송장·번호판·
주소판·통화 소리** 같은 파편 정보가 모여 신상·거주지가 특정되는 **'퍼즐 도싱'**으로 발생한다.
기존 일괄 모자이크는 영상미를 훼손해 사용자가 보안 처리를 기피하게 만들었다.

### 1.2 타겟 고객
- **메인(인플루언서·크리에이터)**: 영상미를 100% 보존하며 개인정보만 자연스럽게 지우는 솔루션에
  비용(구독·크레딧)을 지불할 고가치 고객.
- **서브(일반 대중)**: 중고거래 인증·일상 SNS 업로드 시 유출 예방. 무료 진단 리포트로 유입.

---

## 2. 기술 요구사항 (유지)
1. **YOLOv11·SAM2·STE 모델 배제** — GPU 부하/지연 방지로 미사용 확정.
2. **인페인팅 마스킹** — 텍스트 박스 영역을 주변 픽셀로 자연스럽게 채워 이질감 제거.
3. **단어 단위 Beep 오디오 마스킹** — Whisper `word_timestamps` + NER/정규식으로 핵심 단어의
   ms 단위 구간에만 1000Hz Sine Beep 오버레이(대화 단절 없음).

## 3. ★ 변경된 처리 아키텍처: 5분할 + result.json 중심 (v1 대체)

### 3.1 모듈 구성
| 모듈 | 환경 | 역할 | 산출물 |
|------|------|------|--------|
| `OCR_pipeline_report.py(OCR_Setting.py포함)` | **로컬 CPU** | 시각 PII(OCR) 탐지 | `{stem}_index.json` |
| `colab_pipeline_stt.py` | **Colab** | 음성 STT+PII 탐지 | `{stem}_stt.json` |
| `backend_json_merger.py(실행시 view 코드 자동실행)` | 로컬 | 시각+음성 병합, 마커 생성 | `{stem}_result.json` |
| `pipeline_detail_view.py` | 로컬 CPU | 상세보기 선(先)생성 | `_상세보기.mp4`, `_tracks.json` |
| `colab_pipeline_mask.py` | **Colab GPU** | 선택분만 마스킹 | 최종 마스킹본 |

### 3.2 ★ result.json = 단일 마스터 데이터
모든 후속 단계(리포트·상세보기·선택·마스킹)가 이 한 파일을 기준으로 동작한다.
- `pii_groups[]` (시각): 좌표(bbox/polygons/keyframes), 위험도, `is_selected`
- `audio_pii_groups[]` (음성): 발화 구간(start/end_time_sec), detected_text
- `timeline_markers[]`: 재생바 마커
- **`is_selected`** (초기 false) → 사용자가 켠 항목만 마스킹

### 3.3 ★ 처리 속도·병목 고려
- **시각(로컬)·음성(Colab) 병렬 탐지**로 분석 시간 단축.
- **상세보기 선생성**: merge 직후 미리 만들어 상세보기 클릭 시 지연 0.
- **선택분만 마스킹** + **미리보기는 6초 샘플만** → 불필요 연산 제거.

## 4. ★ 사용자 관점 7단계 흐름

| 단계 | 사용자 행동 | 시스템 동작 | 사용 데이터 |
|------|------------|------------|------------|
| 1 | 파일 업로드 | 청크 업로드·병합 → 분석 Job 생성(크레딧 차감) | 원본 |
| 2 | 진행률 확인 | 시각(로컬)+음성(Colab) **병렬 탐지** → merge → 상세보기 선생성 | index/stt/result.json |
| 3 | **상세보기** 클릭 | 영상/이미지에 탐지된 **모든 PII 박스 표시**(지연 없이 즉시) | result.json + 선생성 mp4 |
| 4 | 가릴 항목 **선택** | 선택 항목 `is_selected=true` 갱신 | result.json |
| 5 | **미리보기** 클릭 | 영상=해당 PII **6초 샘플** 마스킹(가림막 좌우 비교) / 이미지=전체 1장 | result.json + mask(샘플) |
| 6 | **처리 진행** | 원본을 불러와 **선택분(`is_selected=true`)만** 전체 마스킹 | result.json + mask(전체) |
| 7 | **다운로드** | 완료된 마스킹본 미리보기 + 다운로드 제공 | processed_files |

> ★ 미리보기 UX 규칙:
> - **영상**: 각 PII 항목 옆에 개별 미리보기 버튼. 누른 항목만 앞3초~뒤3초(6초) 샘플 생성.
>   전체 선택 수를 모르므로 미리보기 화면엔 "처리요청" 없이 **뒤로가기만**.
> - **이미지**: 1장이라 빠르므로 하단에 단일 미리보기(전체 마스킹). 여러 장이면 샘플 1장 +
>   하단 화살표로 장 넘김. 처리완료/뒤로가기 가능.

## 5. ★ 결과물 제공 방식
- **상세보기**: merge 직후 만들어둔 `_상세보기.mp4`(박스 오버레이)+`_tracks.json`을 그대로 표시 → 대기 없음.
- **미리보기**: 마스킹본과 원본을 좌우 절반씩 합쳐 **중앙 가림막을 좌우로 드래그**해 비교.
- **본처리**: result.json에서 사용자가 켠 ID만 골라 마스킹(예: 5건 중 3건 선택 → 정확히 3건만 처리).
- **최종 다운로드**: 워터마크 적용된 결과물을 보관기간(`expires_at`) 내 다운로드.

## 6. ★ 기존 프론트/백엔드에 맞춘 연결 원칙

> 우리 파이프라인을 기준으로 프론트/백엔드를 바꾸지 않는다. **기존 구조를 최대한 재사용**하고,
> 비효율적인 부분만 신규 개발한다.

| 데이터 | 기존 위치 재사용 | 신규 |
|--------|----------------|------|
| 음성 PII | `detections('voice_pii')` — **이미 적재** | — |
| 시각 PII 요약 | `detections('visual_pii'+bbox)` | `save_pii_result()` 확장 |
| result.json·상세보기·tracks | `analysis_artifacts(stored_path)` — `save_artifact()` 재사용 | — |
| 사용자 선택 | `replacement_actions.is_user_selected` 재사용 | 선택 저장 API |
| 마스킹 작업 | 기존 worker 큐 패턴 재사용(`job_type=mask_*`) | Job 트리거 |
| 최종 결과물 | `processed_files` 재사용 | 조회 API |

> **★ 비효율 회피 판단**: keyframes/polygons 같은 무거운 좌표는 DB 컬럼이 아닌 `result.json`
> 파일(artifact)로 두고, 조회 API는 가벼운 메타만 반환. 마스킹 워커만 파일을 직독한다.
>
> 프론트 mock 5페이지(Report/ReplaceOptions/Preview/Processing/Download)는 레이아웃이 이미
> 완성돼 있어, 위 조회/선택/마스킹 API가 붙으면 **데이터 바인딩만 얹어 자연 연동**된다.
