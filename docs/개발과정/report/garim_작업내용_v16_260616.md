# garim_작업내용_v16_260616

## 작업 개요
- **마스킹 미리보기 버그 및 Colab 워커 에러 수정**: 미리보기 시 마스킹이 전혀 적용되지 않는 문제와, Colab 워커에서 간헐적으로 발생하던 `'NoneType' object has no attribute 'get'` 에러를 동시에 해결했습니다.

## 원인 분석 (디버깅)
1. 프론트엔드(`ReplaceOptions.jsx`)에서 선택된 `pii_id` 목록은 백엔드로 정상적으로 전달되고 있었으며, DB(`replacement_actions`)에도 `is_user_selected = true` 로 정상 저장되고 있었습니다.
2. 하지만 `backend/services/worker.py`의 `get_mask_job_context()` 함수에서 Colab 워커로 전달할 `result.json` 내용을 불러올 때, 병합된 최종 파일인 **`pii_result`**가 아닌 원본 OCR 결과인 **`visual_ocr_json`**을 잘못 조회하고 있었습니다.
3. 이 잘못된 조회로 인해 두 가지 증상이 발생했습니다:
   - **증상 1 (마스킹 안 됨)**: `visual_ocr_json` 파일이 존재하는 경우, 파일 내용은 읽어왔지만 그 안에 개인정보 그룹화 정보(`pii_groups`)가 없어 Colab이 마스킹할 대상을 찾지 못해 원본 이미지를 그대로 반환했습니다.
   - **증상 2 (Colab 에러 발생)**: 해당 타입의 아티팩트가 없거나 파일을 찾지 못한 경우, `result_json` 자체가 `None`으로 넘어가면서 Colab 파이프라인에서 `.get()` 메서드를 호출하다가 `'NoneType' object has no attribute 'get'` 에러가 발생하며 작업이 강제 종료(실패)되었습니다. (첨부해주신 스크린샷의 에러 원인)

## 변경 내용
- **`backend/services/worker.py` 수정**
  - `get_mask_job_context()` 함수 내에서 artifact_type 조회 조건을 `'visual_ocr_json'`에서 `'pii_result'`로 변경했습니다.
  - 이를 통해 Colab 마스킹 워커가 `pii_groups` 배열을 포함한 완전한 분석 리포트를 무조건 정상적으로 전달받게 되어, 선택된 PII 영역에 대해 정확히 마스킹 처리를 수행함은 물론 `NoneType` 에러도 원천 차단되었습니다.
  - 해당 함수의 docstring 설명도 `visual_ocr_json`에서 `pii_result`로 함께 정정하였습니다.

## 검증 계획
- 로컬 또는 Colab 환경의 워커를 재시작한 뒤, 프론트엔드 UI에서 마스킹 체크박스를 선택하고 [미리보기 생성] 시 정상적으로 블러 처리된 결과물이 나타나는지 확인합니다.
- Colab 워커 로그에서 `'NoneType' object has no attribute 'get'` 에러가 더 이상 발생하지 않는지 확인합니다.

## 추가 작업 2: 영상 상세보기 "개별 PII 미리보기" 누락 버그 수정 (260616)

### 증상
영상 상세보기에서 각 개인정보 탐지의 [미리보기] 클릭 시 코랩 워커에서 실패:
```
👁  미리보기: 08a7ec51-..._카드_음성_영상1 (선택: 전체)
  ℹ️  미리보기: 선택된 PII 없음
ERROR:garim-worker:job 처리 실패: 미리보기 생성 실패 — 원본 파일 없음 또는 선택 PII 없음
```

### 근본 원인 (확정)
`analysis_jobs.message` 컬럼이 **두 용도로 충돌**:

| 시점 | 동작 | message |
|------|------|---------|
| ① job 생성 | `create_mask_job`이 `target_pii_id`를 message에 저장 | `target_pii_id:카드번호_1` |
| ② 워커 시작 | `report_progress("file_download")` → `update_job_progress`가 message **덮어씀** | `다운로드 완료…` |
| ③ 컨텍스트 조회 | `get_mask_job_context`가 message 읽음 → **target_pii_id 이미 소실** | → None |

워커 순서가 **②(진행보고) → ③(컨텍스트 조회)** 라서 target_pii_id가 진행 메시지로 덮어써진 뒤 읽힘
→ `selected_pii_ids=[]` → `(선택: 전체)` → `_events_from_groups`가 전부 스킵 → "선택된 PII 없음".

- **DB 실증**: 실패 mask_preview job들의 message가 `target_pii_id:...`가 아닌 진행 메시지로 남아있음을 직접 확인.
  result.json의 `pii_id`(`카드번호_1/2`)와 DB `detections.pii_id`는 정상 일치 → 데이터는 멀쩡, **전달 통로(message)만 손상**.

### 수정 내용 (전용 컬럼 분리 — 콜랩 워커 코드 변경 불필요, 백엔드만 수정)

| 파일 | 변경 |
|------|------|
| `docker/database/init/6_add_target_pii_id_to_jobs.sql` | **신규** — `analysis_jobs.target_pii_id TEXT` 컬럼 추가 마이그레이션 |
| `backend/services/analysis.py` (`create_mask_job`) | target_pii_id를 message가 아닌 **전용 컬럼**에 저장. message는 진행상태 라벨 전용으로 분리 |
| `backend/services/worker.py` (`get_mask_job_context`) | message 파싱 제거 → **target_pii_id 컬럼에서 직접 읽기** |
| 운영 DB | `ALTER TABLE analysis_jobs ADD COLUMN IF NOT EXISTS target_pii_id TEXT` 즉시 적용 |

### 검증 결과
- 가짜 mask_preview job 생성 → 진행보고로 message 덮어쓰기 재현 → `get_mask_job_context` 호출:
  - 수정 전: `selected_pii_ids=[]` (실패)
  - **수정 후: `selected_pii_ids=['카드번호_1']`** ✅ (진행보고 후에도 target_pii_id 보존)
- result.json `pii_id` ↔ DB `detections.pii_id` 일치 확인 → `_events_from_groups` 매칭 정상.
- ※ 검증 중 대화형 Python 서브프로세스가 구글드라이브 절대경로의 파일을 못 읽는 현상이 있었으나,
  이는 Drive 파일스트림 지연 마운트 환경 문제이며 프로덕션 백엔드(uvicorn)는 정상 접근(실패 로그의 source_name 출력이 증거).

### mask 코드 동작 점검 (요청하신 4 시나리오)

| 시나리오 | 코드 동작 | 비고 |
|---------|----------|------|
| 이미지 미리보기 | 선택 PII 마스킹 + 보이는 워터마크(`add_preview_watermark`) | ✅ |
| 이미지 처리진행 | 전체 마스킹 + 안보이는 + (작은 가시) 워터마크 | ⚠️ 스펙은 "안보이는"만 |
| 영상 미리보기 | 탐지구간 6초 클립만 마스킹 + 보이는 워터마크(`watermark_preview=True`) | ✅ |
| 영상 처리진행 | 선택 PII 추적 마스킹 + 안보이는 + (작은 가시) 워터마크 | ⚠️ 스펙은 "안보이는"만 |

→ GPU 인페인팅은 코랩에서 실행되는 구조 정상. 처리진행물의 "작은 가시 워터마크" 포함 여부만 스펙과 미세 차이(추후 확인 필요).

### 후속 검증 필요 (실제 동작)

## 추가 작업 3: 영상 미리보기 가림바 — 원본/마스킹 시간축 불일치 수정 (260616)

### 증상
가림바 비교에서 **왼쪽(원본)은 전체 영상(0초부터)**, **오른쪽(마스킹)은 탐지구간 6초 클립**이라
같은 시점을 비교하지 못함(예: 13초 탐지 → 마스킹은 11~16초인데 원본은 0초부터 재생).

### 원인
- 원본 레이어: `getDetailFileUrl(jobId,"video")` → `_상세보기.mp4` **전체 영상** 서빙
- 마스킹 레이어: 워커 `preview_video`가 만든 **6초 클립**(0초 시작)
- `ComparisonSlider`는 원본 기준으로 마스킹을 동기화하는데, 시간축(전체 vs 6초)이 달라 어긋남.

### 수정 (콜랩 워커 무변경 — 백엔드+프론트만)
원본도 마스킹 클립과 **동일 구간 6초 클립**으로 잘라 서빙 → 양쪽 0초 시작으로 일치.
워커의 구간 계산을 **백엔드가 동일하게 재현**(해당 PII keyframe 최소 frame ± fps×3).

| 파일 | 변경 |
|------|------|
| `backend/controllers/analysis.py` | `_make_detail_clip()` 신규 — result.json의 pii_id keyframe으로 `preview_video`와 동일 구간 계산 → imageio-ffmpeg로 `_상세보기.mp4`를 [lo,hi] 재인코딩 클립 생성(`{영상}_clip_{pii_id}.mp4` 캐싱). `get_detail_file_handler`에 `pii_id` 파라미터 추가 |
| `frontend/src/utils/api.js` | `getDetailFileUrl(jobId, fileType, piiId)` — pii_id 쿼리 추가 |
| `frontend/src/pages/garim/Preview.jsx` | 영상 원본 URL에 piiId 전달 |

### 검증
- 구간 계산이 `preview_video`와 동일함을 코드 리뷰로 확인:
  `anchor=min(keyframe frame)`, `margin=int(fps*3.0)`, `lo=max(0,anchor-margin)`, `hi=min(total-1,anchor+margin)` — 양쪽 클립 모두 [lo,hi]를 0초부터 시작 → 시간축 일치.
- imageio-ffmpeg 번들 ffmpeg 정상 발견, 컨트롤러 문법/함수 정의 확인.
- ※ 실제 파일 클립 생성은 대화형 Python이 구글드라이브 절대/상대경로 파일을 못 보는 환경 제약으로 이 자리에선 미실행. 프로덕션 백엔드(uvicorn)는 정상 접근하므로 실제 동작.
- ⚠️ 미세 차이 가능: `preview_video`는 영상 실제 fps(`cap.get(FPS)`), 백엔드는 `result.json`의 fps 사용 — 둘이 다르면 1프레임 내외 오차. 보통 동일값이라 무시 가능.

### 후속 검증 필요
- 백엔드 재시작 → 영상 개별 [미리보기] → 가림바 좌/우가 **동일한 6초 구간**으로 재생되는지 확인

## 추가 작업 4: 상세보기 박스 픽셀추적 누락 수정 — 모든 keyframe 앵커 (260616)

### 원인
