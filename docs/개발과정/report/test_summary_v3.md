# 🏆 개인정보(PII) 비식별화 및 물리적 합성 파이프라인 개발 종합 요약서

이 문서는 프로젝트 초기 기획 및 요구사항 정의 단계부터, AnyText STE 최초 시도 및 실패, OCR 엔진 선택, YOLO+SAM2 객체 탐지, 개인정보 탐지 엔진, 가짜 데이터 의미적 치환, LaMa 배경 복원, 그리고 최근 피드백을 통해 극상의 퀄리티를 확보한 **"물리적 2중 보호막 및 잉크 블리딩 합성"** 단계까지의 전체 진화 과정을 흐름 순서로 일목요연하게 정리한 종합 보고서입니다.

---

## 1. 📋 프로젝트 개요 및 목표

- **서비스 목적:** 개인정보 보호 자동화 서비스 구축.
- **최종 목표:** 신분증, 우편물, 택배송장 등 다양한 문서 이미지 내 개인정보(PII)를 정확하게 탐지하고, 주변 원본 배경(위조방지 마크, 표 구분선 등)의 어떠한 훼손도 없이 맥락에 맞는 가짜(합성) 데이터로 **감쪽같이** 치환하는 AI 자동화 파이프라인 구축.
- **최우선 타겟:** 실물 촬영 시 훼손 위험이 가장 높은 **'영수증', '택배송장', '신분증'** 3대 문서 이미지를 기준으로 기능 검증 및 화질 고도화 진행.
- **실행 환경:** Google Colab (GPU 사용), 모델 가중치는 Google Drive 마운트 경로(`/content/drive/MyDrive/final_PJ_model/`) 활용.
- **출력물:** `/content/drive/MyDrive/final_PJ_model/output_file/` 폴더에 결과 이미지 저장.

## 2. 🚀 전체 개발 진화 흐름 및 단계별 구현 내역

### 📍 [Phase 0] AnyText STE 최초 시도 및 실패 → 파이프라인 전환

#### 0-1. 최초 접근: AnyText 기반 단일 파이프라인
- 초기 구상은 YOLO/SAM2로 개인정보 영역을 탐지한 뒤, **AnyText(Scene Text Editor) 모델** 하나로 OCR·배경 지우기·가짜 텍스트 합성을 일괄 처리하는 End-to-End 파이프라인.
- 모델 가중치는 Google Drive의 `STE/` 폴더에 보관.

#### 0-2. 핵심 오류 발생 및 원인 분석
```
⚠️ AnyText 추론 중 예외 발생: expected scalar type Float but found Half
⚠️ AnyText 합성 실패 (해당 부분 원본 유지)
```
- **원인:** AnyText 모델이 `fp16(Half)` 정밀도로 로드되어 있는 반면, 입력 텐서가 `fp32(Float)`로 전달되어 발생한 dtype 불일치.
- **결과:** 6개 전 이미지에서 합성 단계가 전부 실패. 개인정보 탐지(🚨)와 LLM 치환(🔄)은 성공했으나 이미지에 반영되지 못함.

#### 0-3. AnyText2 업그레이드 검토 및 한국어 지원 한계 분석
- `https://github.com/tyxsspa/AnyText2` 확인: AnyText2(AnyText2XL 포함)가 출시되었으나 당시 미완성 상태.
- **한국어 지원 근본 한계 분석:**
  - AnyText 내부의 `ocr_recog/`, `ocr_weights/` 폴더 조사 결과, 인식 모델(PP-OCRv3)이 중국어·영어 글리프만으로 학습됨.
  - `font/` 폴더도 한국어 폰트 미포함.
  - 완전한 한국어 지원을 위해서는 PP-OCRv3를 한국어 OCR 데이터로 **재학습**하고, 전체 AnyText 디퓨전 모델을 한국어 포함 데이터로 **재학습**해야 하며 — 현실적으로 불가 판단.
- **결론:** AnyText STE 의존도를 낮추고, OCR·배경 복원·폰트 합성을 **각 전문 컴포넌트로 분리**하는 파이프라인으로 전환.
- `down.py` 스크립트 작성: AnyText2 모델 가중치를 `STE2/` 폴더로 다운로드하는 스크립트 구현 (향후 AnyText2 전환 대비).

### 📍 [Phase 1] OCR 엔진 선택 및 고도화

#### 1-1. OCR 엔진 비교 평가 여정

| 엔진 | 평가 결과 |
|------|----------|
| **EasyOCR** | 초기 사용. 한국어 인식 가능하나 좌표 정밀도 낮음 |
| **Google Cloud Vision API** | 바운딩 박스 정밀도 우수. 초기 Phase에서 채택 |
| **PaddleOCR** | AnyText와 달리 GPU 충돌 없이 작동 확인. 무료·오픈소스 |
| **Naver Clova OCR (HYPERCLOVA)** | **한국어 OCR 최고 성능.** `.env` 파일의 개인 API 키 연동 테스트 진행. 유료(비용 모니터링 필요) |
| **Tesseract** | 로컬 서버 설치형. 무료·무제한. 한국어 완벽 지원은 아니나 YOLO 탐지 후 ROI 크롭 활용 시 충분히 활용 가능 |
| **STELLAR / Gemini Vision API** | 검토 대상. Gemini Vision은 API 호출 시 개인정보 외부 노출 가능성·유료 비용·횟수 제한 검토 필요 |

- **최종 채택:** 한국어 정확도가 가장 높은 **Naver Clova OCR**을 메인으로, PaddleOCR을 폴백으로 사용하는 구조.
- **ImportError 해결:** `huggingface_hub.errors`에서 `OfflineModeIsEnabled` import 실패 오류 → 라이브러리 버전 호환성 수정.

#### 1-2. JSON 좌표 캐싱 시스템 도입
- **문제:** 합성 파라미터 튜닝 시 동일 이미지에 대해 매번 OCR API를 호출하면 비용·시간 낭비.
- **해결:** OCR 결과(텍스트 + 바운딩 박스 좌표)를 `ocr_test_output/` 폴더에 **이미지별 JSON 파일로 캐싱**.
- 이후 합성 테스트 시에는 JSON 파일을 직접 로드하여 OCR API 재호출 없이 반복 합성 가능.

#### 1-3. 텍스트 추출 및 바운딩 박스 정밀화
- 이미지 내 텍스트와 정확한 바운딩 박스(Bounding Box) 좌표를 단어/줄 단위로 실측 추출 및 군집화.
- 각도가 틀어진 텍스트 탐지를 위해 수직/수평 직선 외 기울어진 객체 각도까지 고려하는 방향으로 개선.

#### 1-4. 영상(Video) 처리 범위 확장 논의
- 이미지 처리와 동일한 원리로 **영상도 프레임 단위(frame-by-frame) 처리** 시 구현 가능함을 확인.
- 웹 서비스에서 이미지 + 영상 모두 개인정보 비식별화 제공 방향으로 범위 설정.

### 📍 [Phase 2] YOLO + SAM2 객체 탐지 파이프라인

#### 2-1. YOLO 탐지 단계
- 입력 이미지 전체에 YOLO 모델을 적용하여 개인정보가 존재할 가능성이 높은 **영역(Region of Interest, ROI)** 을 객체 단위로 탐지.
- 탐지 성공 시 → SAM2 정밀 마스킹 진행.
- 탐지 실패 시 → 원본 이미지 전체를 대상으로 OCR 수행(폴백).

#### 2-2. SAM2 정밀 마스킹
- YOLO가 탐지한 ROI에 **SAM2(Segment Anything Model 2)** 를 적용하여 픽셀 단위의 정밀 세그멘테이션 마스크 생성.
- 마스킹된 영역에서 텍스트가 감지되지 않을 경우 → 전체 원본 이미지 OCR 재시도.

#### 2-3. OCR 탐지 단계 처리 예시
```
📐 이미지 크기: 740x475
🎯 YOLO 탐지 성공: 1개 객체 발견! (SAM 2 정밀 마스킹 시작)
🔍 텍스트 추출 중...
✅ 24개 텍스트 박스 감지
```

### 📍 [Phase 3] 텍스트 정밀 추출 및 개인정보(PII) 탐지 엔진

#### 3-1. 정규표현식(Regex) 엔진 고도화
- 주민등록번호, 휴대폰 번호, 주소, 카드번호, 계좌번호, 이메일, 여권번호 등 핵심 개인정보 패턴 완벽 탐지.
- 이름·등기번호·사업자번호·일반 전화번호는 정규식 대상에서 명시적으로 제외.

#### 3-2. 주소 Greedy(탐욕적) 병합 및 2차 필터링
- 주소 중간에 띄어쓰기나 무의미한 숫자가 끼어들어 텍스트가 토막 나는 현상 방지.
- 한 줄 전체를 탐욕적으로 쓸어 담은 뒤 유효한 주소 접미사(길, 로, 동)와 숫자만 Python 단에서 정밀 검증하는 **'2차 정제 필터'** 구축.

#### 3-3. 오탐(False Positive) 처리
- 개인정보가 아닌 텍스트(예: 상품명, 바코드, 일련번호)가 개인정보로 오탐되는 케이스 지속 식별 및 필터 개선.
- OCR-전용 테스트 단계(`ocr_test.py`)를 합성 파이프라인과 분리하여, 탐지 정확도를 먼저 검증한 뒤 합성으로 진행하는 워크플로우 정립.

### 📍 [Phase 4] 맥락 보존형 가짜 데이터(Fake Data) 생성 엔진

#### 4-1. 정규식 기반 1:1 구조 치환
- 주민번호나 전화번호의 자릿수, 구분자(대시 `-`, 점 `.`, 공백 등)의 형태를 원본 그대로 유지하며 무작위 치환.

#### 4-2. Gemini LLM 기반 의미적 치환 엔진
- 단순한 가비지 텍스트가 아닌, 실제 존재하는 듯한 자연스러운 **한국형 주소와 명칭**을 생성하기 위해 Gemini Flash 모델 연동.
- 의미와 맥락을 보존한 가짜 데이터 생성.
- **한계 확인:** LLM이 다중 줄 주소(`L5: 서울시 노원구 동일로 228길 / L6: 청수빌딩 204호`)를 하나의 문자열로 합쳐 치환할 때, 두 번째 줄 정보(건물명)를 누락하거나 잘못된 형태로 생성하는 케이스 발생 → 분리 처리 로직 강화.

### 📍 [Phase 5] 초정밀 픽셀 타겟팅 기반 배경 복원 (LaMa 인페인팅)

#### 5-1. LaMa 도입 배경
- 초기 단순 사각형 덮어쓰기 방식 → 배경 패턴이 뭉개지고 표의 격자선이 끊기는 문제 발생.
- **LaMa(Resolution-robust Large Mask Inpainting)** 채택: 신분증 배경의 반복 격자무늬, 지구본 홀로그램 선 등을 주변 패턴 참조로 끊김 없이 복원 가능.

#### 5-2. Otsu 이진화 잉크 마스킹
- 단순 사각형 전체 지우기가 아닌, 글씨의 **검은색 잉크 픽셀만** 정밀하게 마스크로 추출 후 팽창.

#### 5-3. LaMa 인페인팅 적용
- 마스킹된 텍스트 영역을 지우고 주변 배경의 그라데이션 및 질감 패턴을 참조하여 끊김 없이 매끄럽게 복원.

### 📍 [Phase 6] 동적 폰트 스캔 매칭 및 cv2.warpPerspective 3D 합성

#### 6-1. 원본 글씨 특징 스캔
- 폰트의 굵기(Stroke Thickness), 잉크 색상(RGB), 장평(Aspect Ratio), 선명도를 사전 추출.

#### 6-2. 150종 폰트 정밀 수학적 매칭
- 보유한 150여 개의 서체 파일 전체의 '글자 밀도'와 '장평 비율'을 수학적으로 비교하여, 오차가 0에 수렴하는 최적의 폰트를 동적으로 선정.

#### 6-3. 3D 기하 왜곡(Warping) 합성
- 2D 평면 합성의 한계를 극복하기 위해 `cv2.getPerspectiveTransform` 및 `cv2.warpPerspective`를 사용하여 이미지 촬영 구도에 맞게 3D 비틀기 정밀 합성.

## 3. 🛡️ [핵심 보완] 최종 퀄리티 확보를 위한 물리적 고도화 단계 (Phase 7 - 최신)

최근 발생했던 **"배경 및 인접 글자 훼손, 표 격자선 끊김, 점자 같은 가짜 잉크 질감"** 문제를 완벽하게 해결하기 위해 최종 도입된 물리적 고도화 및 2중 보호 패키지 구현 내역입니다.

### ① 🚨 표의 구분선(가로/세로 격자선) 및 경계 그림자 철통 보존
- **문제점:** 지우기 마스크가 확장되면서 표의 얇은 회색 격자선이나 스탬프 종이 경계선의 그림자가 함께 지워져 표가 끊어지는 현상 발생.
- **해결책:** 수평/수직 구조선 모폴로지 기법(`MORPH_OPEN`)을 연동하여 ROI 내의 **가로/세로 격자 구분선을 실시간 감지**. 해당 구조선 영역(`grid_mask`)을 지우기 마스크에서 **원천 차감**하여 단 1픽셀도 훼손되지 않도록 100% 보존.

### ② 🛡️ 비대상 원본 텍스트 2중 격리 보호막 (사방 3px 마진)
- **문제점:** 합성 대상 텍스트 바로 옆에 바짝 붙어 있는 보존 대상 글자("성명", "전화", "1588-1300" 등)가 마스크 팽창으로 인해 획의 끄트머리가 허옇게 깎여나가는 문제.
- **해결책:** 이미지 전체에서 지우지 말아야 할 모든 비대상(Non-PII) 박스 좌표를 사전 수집. 지우기(`erase`) 및 렌더링(`render`) 마스크 연산 시 이 비대상 영역에 **사방 3px의 안심 마진**을 더하여 강제로 마스크 값을 차단(`bitwise_and not`). 인접한 원본 텍스트의 훼손도가 수학적으로 **0.0000%** 에 수렴.

### ③ ✒️ 잉크 번짐(Ink Bleeding) 재현 및 선명도 대비 강화를 통한 실물 질감 복원
- **문제점:** Salt & Pepper 구멍 노이즈가 디지털 점자 가루처럼 숭숭 비어 보이는 심각한 이질감 유발.
- **해결책:**
  - 거친 픽셀 노이즈를 **전면 삭제**.
  - **선명도 대비 강화:** 마스크 값을 `1.45배` 증폭시켜 획 내부를 짙게 채움.
  - **Ink Bleeding(잉크 번짐):** 잉크젯 프린터나 도장 잉크가 종이 섬유 결을 타고 미세하게 스며 번져나가는 현상을 재현하기 위해 연한 가우시안 블러(`sigma=0.4`)를 결합해 Multiply 융합.

### ④ 💡 Specular Highlight(반사광) 보존 및 조건부 Grain 주입
- **반사광 복원:** 신분증 등 플라스틱 표면의 빛 반사 영역(밝기 >= 220)을 추출한 뒤 합성 완료된 텍스트 위로 다시 통과시켜 글자가 카드 필름 내부에 박혀 있는 자연스러운 3D 입체감 완성.
- **조건부 Grain:** 표면 분산(variance)이 300 이하인 매끄러운 플라스틱 카드는 종이 노이즈를 제거하고, 일반 종이(송장)일 때만 신규 글씨 획 영역 내부에 국소적으로 종이 고주파 노이즈를 입혀 이질감 완전 상쇄.

## 4. ⚠️ 현재 직면한 한계점 및 다음 스텝 예정 (미구현 과제)

물리적 질감 결합(반사광, Grain, 구분선 보호) 고도화에도 불구하고, 실제 프로덕션 수준의 상용 서비스를 위해 **반드시 추가 해결해야 하는 근본적 한계점 및 구현 예정 스텝** 입니다.

### 1. 🧼 얼룩(배경 찌꺼기) 완벽 제거 (마스킹 방식 교체)
- **현재 한계:** OpenCV 이진화 기반의 '잉크 픽셀'만 정밀 타겟팅해 지우는 방식은 글자 외곽의 미세한 안티앨리어싱 찌꺼기(얼룩)를 남겨 배경 복원 후에도 잔상 발생.
- **다음 스텝:** 개별 단어의 **바운딩 박스(사각형) 또는 다각형(Polygon) 영역 전체를 통째로 마스킹**하여 배경의 노이즈와 얼룩을 한 픽셀의 오차도 없이 완벽하게 날려버리는 방식으로 교체 예정.

### 2. 🔍 주소 합침/분리 정밀도 극대화 (2차 필터 Truncate 강화)
- **현재 한계:** 탐욕적(Greedy) 주소 병합 시 주소 뒤에 연달아 나오는 무관한 숫자 덩어리(금액, 바코드 등)나 영문 시리얼 코드가 주소의 일부로 합쳐져 마스킹 템플릿 붕괴 오류 발생.
- **다음 스텝:** 병합된 주소 문자열 중간에 주소와 무관한 완전히 독립된 긴 숫자 덩어리(5자 이상)나 영문 대문자 혼합 코드가 등장하여 흐름이 끊기면, **그 즉시 탐지를 중단(Truncate)** 하고 앞부분까지만 주소 박스로 확정하는 정밀 2차 필터 탑재 예정.

### 3. 🎨 폰트 이질감(가짜 느낌) 완벽 제거 (단어별 개별 3D 합성)
- **현재 한계:** 여러 단어로 이루어진 긴 문장을 통째로 하나의 일직선 캔버스에 그려 늘이거나 비트는 방식(`cv2.warpPerspective`)은 폰트의 고유 장평이 찌그러지거나 일그러지는 주범.
- **다음 스텝:** 가짜 텍스트를 단어별로 미세하게 쪼갠 뒤, 원본 개별 단어 박스의 고유한 높낮이, 장평, 기울기에 1:1 매칭하여 **단어 하나씩 독립적으로 3D 투시 변환 합성(Piecewise Word Warp Rendering)** 을 수행하여 실제 인쇄물과 구분 불가능한 기하 품질 확보 예정.

### 4. 🔤 AnyText2 한국어 확장 (장기 과제)
- **현재 한계:** AnyText(구버전 및 v2 모두) 한국어 글리프 학습 미포함. `expected scalar type Float but found Half` dtype 불일치 문제로 합성 전면 실패.
- **다음 스텝:** PP-OCRv3 한국어 OCR 데이터 재학습 + AnyText 디퓨전 모델 한국어 포함 파인튜닝. 또는 `down.py`를 통해 AnyText2 가중치 다운로드 후 `STE2/` 경로로 교체하여 재시도.
