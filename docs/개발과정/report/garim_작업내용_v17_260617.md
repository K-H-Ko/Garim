# garim 작업내용 v17 (2026-06-17)

## 주제: 이미지 인페인팅 잔상/덜 지워짐 개선 시도 → 롤백 + 정확한 원인 재진단

### 배경 (증상)
미리보기 마스킹 결과에서 ① 검은 텍스트 덜 지워짐(스탬프 위 글씨), ② 투명 잔상(전화번호),
③ 송장 경계선 어긋남이 관찰됨.

### 1차 시도 (❌ 실패 — 롤백함)
`synthesize_pii_region()` 일반 PII 마스크를 키우는 방향으로 변경:
- adaptiveThreshold + Otsu OR 결합 / morphologyClose / dilate 1→2회 / MASK_COMPLEXITY_THRESH 0.12→0.06

**실패 사유 (사용자 반려):** 마스크를 전역으로 키우자 **주변 배경·표 선이 전부 뭉개짐**.
사용자 최우선 원칙("주변 훼손 없이 텍스트만 타이트하게")을 정면으로 위반. 일부 글자는 여전히 미삭제.
→ 마스크 확대 접근은 역효과로 판명. **전 변경 원복(코드 원래 상태로 복구).**

### 정확한 원인 재진단 (좌표 분석)
실데이터(택배송장1, 684×512)의 pii_groups 좌표 확인 결과:
- **스탬프 위 텍스트도 PII 박스는 정상 존재** (전화번호_1 y[6~74], 주소_1 단어 6개 y[41~111]). 박스 누락 아님.
- 그런데 스탬프 흰 종이에 **흐리게 비친(저대비) 글자**라 `adaptiveThreshold`가 잉크를 거의 못 잡음
  → thin_mask가 비어 **박스 안인데도 덜 지워짐**.
- 하단 송장 주소_2는 박스가 한 덩어리(x[132~341])라, 통째로 채우면 표 선까지 훼손됨.

### 결론 / 올바른 방향 (미적용 — 동의·검증 대기)
- 마스크를 전역으로 키우지 말 것(주변 훼손).
- 흐린 글자는 "**박스 내부에서 잉크가 거의 안 잡힌 단어에만** 박스 채우기로 폴백", 선명한 글자는 기존 타이트 마스크 유지 → 주변 보존 + 흐린 글자 제거 양립.
- 큰 덩어리 박스(주소_2)는 채우기 제외(표 선 보호).
- 잔상은 마스크 확대가 아닌 **LaMa 인페인팅 품질**로 해결.

### 검증 한계 (중요)
- 로컬에 LaMa·simple_lama_inpainting 미설치(numpy 빌드 실패) → 로컬은 Telea fallback이라 **잔상 검증 부정확**.
- 실제 검증은 **Colab/GPU + LaMa** 환경에서 수행해야 함.

### 산출물
- 코드: 원래 상태로 복구 완료(변경 없음)
- 검증용 임시파일(_test_mask_quality.py, _mask_compare.jpg 등) 삭제 완료

## 추가 작업 2: OCR region(객체) 분할 — 배경색 + 각도 결합 (`OCR_Setting.py`)

### 배경 (문제)
우편물3 등에서 **송장 라벨 텍스트 + 손글씨 주소 + 우표가 하나의 거대 region으로 묶여**,
정규식 주소 파싱 시 "경북 영덕군 영해면 ... 2014/09/26 766-811즉납 4주8길18 ..."처럼 섞임.
원인: `build_regions`가 송장 라벨을 독립 객체(contour)로 못 잡아(mode="angle+gap only"),
각도·위치가 비슷한 박스들을 한 구역으로 병합.

### 사용자 설계 (객체 분리 판단 로직)
- 배경/주변 색이 다르면 → 다른 객체 (송장 흰 라벨 vs 갈색 박스)
- 색이 같아도 각도가 다르면 → 다른 객체 (겹친 A4 2장) ← 기존 각도 조건이 이미 처리
- → 색·각도 둘 다 만족해야 같은 객체로 병합

### 변경 내용 (`OCR_Setting.py` 4곳)
1. 설정값 추가: `BG_COLOR_ENABLED=True`, `BG_COLOR_THRESH=22.0`, `BG_RING_PAD_RATIO=0.3`
2. 헬퍼 신설: `_bgr_to_lab`, `_color_dist`(LAB 색거리), `_box_bg_color`(박스 외곽 링의 대표 배경색)
3. `cluster_angle_gap` 병합 조건에 추가: 두 박스 배경색 LAB 거리 > 임계면 병합 차단(각도 조건과 AND)
4. `build_regions`에서 박스마다 `bg_color` 부여

### 검증 (우편물3, 실데이터 OCR)
- 임계 스윕: 30=다묶임 / **22=송장↔손글씨 분리(균형점)** / 16↓=송장 라벨 과분리
- THRESH=22 결과: 손글씨 주소(경북 영덕군 영해면/제주옹길18/이경희/766-811)가
  송장 라벨(2014/09/26·17004-0282-4076·등기 이경희)과 **별도 region으로 분리** 확인(시각화).

## 추가 작업 3: 단어별 타이트 박스 — 회전 사각형 → 잉크 포락선 다각형 (`OCR_Setting.py`)

### 배경 (문제)
OCR 시각화에서 단어 박스가 헐겁고, 종이가 휘면(곡면 송장) 글자 윗줄이 곡선인데
박스는 직선이라 양 끝이 뜨거나 가운데가 잘림. 목표는 단어마다 텍스트에 딱 맞는
(휨까지 따라가는) 타이트한 다각형 좌표.

### 원인
`_split_and_snap_to_words`가 단어별 최종 박스를 `cv2.minAreaRect`(4점 회전 사각형)로 생성.
회전 사각형은 기울기 1개만 표현 → 종이 휨(곡선)을 못 따라감.

### 1차 시도 (헐거움 — 사용자 반려)
모든 단어를 잉크 포락선 다각형으로 만들고 minAreaRect+6px 확장 폴백 사용.
→ 결과가 타이트하지 않고, 불필요하게 전부 다각형. minAreaRect가 잉크 외곽 점으로
   자체 각도를 잡아 줄 방향과 어긋나고 +6 확장이 헐거움을 키움.

### 2차 (현재 — 사용자 설계 반영)
"기본은 직각 4점 박스, 휘어서 상/하/좌/우 여백이 불균일한 단어만 다각형" + 타이트화.
글자 진행 방향(PaddleOCR 줄 방향 line_dir)에 정렬된 (u,v) 좌표계에서 잉크 범위에 딱 맞춤.

### 변경 내용 (구역·색·각도·문장조립 로직은 일절 미수정 — 박스 생성부만 교체)
1. 설정값: `WORD_BOX_PAD=1.5`(여백 px, +6→1.5로 타이트화), `WORD_ENVELOPE_COL_PX=8.0`,
   `WORD_STRAIGHT_MIN_TOL=2.0`, `WORD_STRAIGHT_RATIO=0.12`(직선 판정 허용 잔차=글자높이×0.12)
2. `_word_box(all_pts)` 헬퍼 신설(기존 `_word_poly` 대체):
   - (u,v) 투영 후 기본 `_rect_box()`=줄 방향 정렬 타이트 4점.
   - 칸별 상/하 잉크 끝(envelope)을 1차 직선 근사 → 최대 잔차 측정.
   - 상·하 모두 잔차 ≤ tol(여백 균일) → **4점 사각형 유지**.
   - 잔차 초과(종이 휨) → 그 단어만 **칸별 끝점 다각형**.
   - 점<8개/폭<6px면 사각형. (None 폴백 제거 — 항상 좌표 반환)
3. 단어 1개/여러 단어 분기 둘 다 `_word_box` 호출로 단순화(minAreaRect+convexHull 제거).

### 효과 / 호환성
- 대부분 단어는 줄 방향 정렬 타이트 4점, 휜 단어만 다각형 → 요구 충족.
- `vertices`는 기존 시각화(polylines)·JSON에서 그대로 소비 → 하위 파이프라인 무수정.
- 각도값은 하위 `_to_boxes`가 꼭지점으로 `minAreaRect` 산출 → 계속 정상.
- 타이트/곡선전환 강도는 `WORD_BOX_PAD`/`WORD_STRAIGHT_RATIO`/`WORD_ENVELOPE_COL_PX`로 조정.

### 3차 (최종 — 단순화 확정)
2차(직선/곡선 판정 + 잉크 외곽 점 기반)도 결과가 헐겁고 불안정 → 사용자: "그냥 여백 거의
타이트하게, 각 단어별로만 박스". 곡선/다각형/잉크 윤곽 추적을 **전부 제거**하고 단순화.

**최종 로직 (`_split_and_snap_to_words` 전면 재작성):**
1. PaddleOCR 줄(line) 4점 박스를 입력으로 받음.
2. 글자 폭 가중치(한글2/ascii1.1/공백0.6)로 **단어별 가로 비율 구간[sr,er]** 계산.
3. 줄 ROI에서 잉크 마스크(adaptiveThreshold + 폴리곤 클립 + OPEN 잡음제거) 1회 생성,
   잉크 픽셀을 줄 방향 (u,v) 좌표로 보관.
4. 단어별로: 비율 구간을 픽셀 u구간으로 환산 → 그 구간 잉크로 **가로(u)·세로(v)를
   실제 잉크 범위에 좁혀** 타이트화 → 줄 방향 정렬 **직각 4점 박스** 생성(여백 `WORD_BOX_PAD`).
5. 잉크 부족 단어는 줄 박스 높이 그대로 사용(폴백).

- 설정: `WORD_BOX_PAD=1.5`만 유지. (WORD_ENVELOPE_*, WORD_STRAIGHT_* 전부 제거)
- `_word_box`/`_word_poly`/minAreaRect+convexHull/곡선판정 **전부 삭제**.

### 검증 (3차)
- `ast.parse` 문법 정상, 잔여 참조(_word_box/_word_poly/WORD_ENVELOPE/WORD_STRAIGHT) 0건 확인.
- 각도값은 하위 `_to_boxes` minAreaRect로 계속 산출(정상).

### 4차 (최종 — 잉크 갭 + 구두점 분할)
실데이터(택배송장1) text 출력 분석에서 **근본 원인 발견**: PaddleOCR이 한글 단어 사이
공백을 안 넣고 `서울시노원구동일로228길`처럼 한 덩어리로 반환 → 텍스트 공백 분할 무력.
사용자 지시: 실제 글자 사이 빈 공간(잉크 갭)으로 단어를 끊고, `010.4201.7478`는 점이
이어져 갭으로 안 끊기니 구두점(. , -)으로 추가 분할(구두점은 앞 토큰에 붙임).

**최종 로직 (`_split_and_snap_to_words` 재작성):**
1. 줄 ROI 잉크를 진행방향(u축) 1px 칸으로 투영 → 잉크 run/gap 추출.
2. gap > 글자높이 H × `WORD_GAP_RATIO`(0.45)면 단어 경계로 분할, 작으면 병합
   (자모/음절 간격은 유지, 단어 공백만 끊음) → PaddleOCR 무공백 한글 대응.
3. 인식 텍스트를 글자폭 비율로 각 시각단어에 라벨 분배(구역 조립 시 원복되므로 안전).
4. 라벨의 . , - 에서 추가 분할(`010.4201.7478`→`010.`/`4201.`/`7478`).
5. 각 토큰을 실제 잉크 범위(u/v)로 좁혀 줄 방향 정렬 타이트 4점 박스.

- 설정: `WORD_BOX_PAD=1.5`, `WORD_GAP_RATIO=0.45` 추가.

### 검증 (4차)
- `ast.parse` 문법 정상.

### 5차 (실데이터 진단 + 임계 보정)
사용자: 텍스트 결과상 단어가 붙어 보임. **실측 진단(택배송장1 JSON/이미지 ROI 직접 분석):**
- 박스는 이미 단어별 분리됨(Zone2 스탬프: 서울시/노원구/동일로/228길 각각 별도) — 정상.
- `.txt`는 `assemble_text`가 박스를 한 줄로 재조립해 공백 없이 출력 → "붙어 보임"(표시 문제).
- 빽빽한 인쇄 슬랜트 줄(Zone13 `서울시노원구동일로228길35`)은 미분할:
  기울어진 글자가 u축 투영 시 겹쳐 단어 사이 빈 공간(0 잉크)이 사라짐.
- **원인:** `col_thr=0.15*peak`가 너무 낮아 모든 칼럼 점유 → run 1개 → 분할 불가.

**사용자 아이디어 반영(상대 갭):** 옆 글자 간격 비교 = `word_gap=max(0.45×H, 갭중앙값×1.8)`.
이미 적용돼 있었고, **칼럼 점유 임계만 0.15→0.30으로 상향**(신규 상수 `WORD_COL_THR=0.30`).

**파라미터 스윕 실측(택배송장1):**
| col_thr | 빽빽폼줄 | 노원구 | 228길 | 청수빌딩 |
|---|---|---|---|---|
| 0.15 | 1(미분할) | 1 | 1 | 1 |
| **0.30** | **3분할** | 1 | 1 | 1 |
| 0.40 | 1 | 3과분할 | 1 | 3과분할 |
→ 0.30이 단일단어 보존 + 부분 분할 균형점으로 확정.

**한계(정직 보고):** `동일로228길35`처럼 글자간격이 균일·빽빽한 구간은 단어/글자 갭 차이가
거의 없어 완전 분할 불가. 단일단어 과분할을 피하려면 부분 분할이 안전 한계.

### 검증 (5차)
- `ast.parse` 문법 정상. 임시 검증 스크립트 전부 삭제.
- 재실행은 PaddleOCR 환경 필요(로컬 미설치). 조정: `WORD_COL_THR`(끊김 정도), `WORD_BOX_PAD`(여백).

### 6차 (중복 박스 제거 — 주민번호 줄 깨짐 해결)
신분증2(주민등록증)에서 사용자 보고: 같은 글자 중복 추출 + 주민번호 탐지 실패.
**실측 진단(신분증2 JSON):**
- `주민등록증`(전체)와 `주민`/`등`/`록등`(조각), `용길동()`와 `길동`+`()`가 **중복 추출**
  (1차 OCR vs 2.0x 고배율 재OCR가 같은 글자를 다르게 잡아 `_merge_hires`가 다 남김).
- 이름 줄의 중복 `()`가 주민번호 줄에 끼고, `()`↔`2111111` x겹침으로 `2111111`이 다음 줄로
  밀려 `assemble_text`가 `111111- ()` / `2111111`로 분리 → 정규식 `\d{6}-\d{7}` 깨짐.

**해결: `_dedup_boxes` 신설 + `ocr_extract_boxes` 끝에서 호출 (assemble_text·region 로직 무수정).**
- 규칙: 작은 박스가 큰 박스와 ①공간 겹침(작은쪽 면적 기준 50%↑) ②글자가 큰 박스에 포함
  (char 60%↑) ③각도 유사(20° 미만)일 때만 제거.
- ②로 숫자(주민번호)는 인접 줄과 bbox 겹쳐도 보존, ③으로 세로(80°) 박스 보존.

**실측 검증(신분증2 전역):** 29→24박스, 제거={주민,등,록등,길동,()}만. `111111-`,`2111111` 보존.

## 추가 작업 4: 단일 글자가 혼자 region으로 분리되는 문제 — 각도 계산 수정 (`OCR_Setting.py`)

### 배경 (문제)
택배송장1에서 `네이처사이드`의 `사`, 스탬프 전화번호의 `.`/`7` 등 **단일 글자가 같은 줄인데
혼자 다른 region으로 분리**됨. `BG_COLOR_THRESH`를 22→27로 올려도 그대로 → 배경색 원인 아님.

### 원인 (실측 진단, OCR JSON 꼭지점 분석)
- `_to_boxes`의 각도 계산이 `cv2.minAreaRect` 사용.
- 단일 글자는 박스가 **정사각형**이라 minAreaRect가 가로/세로를 구분 못 해 **90° 뒤집힌 각도** 반환
  (예: `사` 84.6°, `.` -78.4° — 실제 줄 각도는 -5°/+12°).
- 줄 각도와 90° 차이 → `cluster_angle_gap`의 `ANGLE_THRESH(10°)` 초과 → 혼자 region 분리.
- 반면 세로쓰기(`보내는분` 87°, `밤는분` 76°)는 박스가 길쭉해 minAreaRect가 정상.

### 해결 (`_to_boxes` 각도 계산만 수정)
- 박스가 **세로로 길쭉할 때만(bbox h > w×1.7)** minAreaRect 장축 각도 사용(세로쓰기 보존).
- 그 외(가로 단어·정사각 단일글자)는 **윗변 tl→tr(읽기 방향) 각도** 사용.
  박스는 [tl,tr,br,bl] 순으로 줄 방향에 맞춰 생성되므로 윗변 각도가 단일 글자도 안정적.

### 검증 (실측 꼭지점 적용)
| 텍스트 | w×h | 기존(minAreaRect) | 신규규칙 | 정답 |
|---|---|---|---|---|
| 사 | 14×16 | 84.6°(분리) | -4.4° | -5° ✓ |
| . | 17×25 | -78.4°(분리) | 14° | +12° ✓ |
| 보내는분 | 32×90 | 87° | 87°(유지) | 90° ✓ |
| 밤는분 | 56×121 | 76° | 76°(유지) | 76° ✓ |
- `ast.parse` 문법 정상. 임시 진단 스크립트 삭제.
- 단일글자가 줄 각도를 갖게 되어 angle 장벽 제거 → 이웃과 병합 가능(나머지 색/높이/간격 조건 충족 시).
- 재실행은 PaddleOCR 환경 필요(로컬 미설치).

## 추가 작업 5: 로컬 복잡도 기반 구역 분리 + 각도 안정화 + 주소 토큰 결합 (`OCR_Setting.py`, `OCR_pipeline_report.py`)

### 5-1. 로컬 복잡도 구역 분리 (`OCR_Setting.py`)
- **문제**: 우편물3에서 송장 라벨(바코드)과 손글씨 주소가 색이 거의 같아(LAB ~20, 박스가 라벨보다 더 밝음) 배경색만으로 분리 불가. 흰색/엣지 사각형 검출도 실패(라벨과 박스 색·밝기 유사).
- **해결**: 사용자 설계(적응형 다중단서) 반영 — `_box_complexity()`(박스 주변 엣지밀도) 신설, `cluster_angle_gap`에 "복잡도 차이 > `COMPLEXITY_THRESH`면 병합 차단" 조건 추가. 색이 비슷해도 바코드(고복잡도)↔손글씨(저복잡도)로 분리.
- **실측 임계값**: 스윕 결과 `COMPLEXITY_THRESH=0.06`에서 즉납·바코드가 주소(경북영덕/군영해면/4주8길18)와 분리됨(0.07↑은 섞임). 설정값 0.06.
- 신규 상수: `COMPLEXITY_ENABLED=True`, `COMPLEXITY_THRESH=0.06`, `COMPLEXITY_PAD_RATIO=1.0`.
- `BG_COLOR_THRESH`는 드리프트(25)→검증값 **22로 복원**.

### 5-2. 각도 안정화 — 회전 사각형 장단축비 사용 (`_to_boxes`)
- **문제**: "북구"의 `구`(8×18) 등 좁은 단일 글자가 `minAreaRect`로 90° 뒤집힌 각도(-88°)를 받아 혼자 region으로 분리. axis 비율(2.25)은 기울어진 세로글자(밤는분 2.16)보다 커서 구분 불가.
- **해결**: 판정을 **회전 사각형 장단축비(기울기 무관)** 로 변경. `rect_aspect ≥ 2.5`(확실한 긴 라인)만 minAreaRect 각도, 그 외 짧은/정사각/좁은 글자는 윗변(읽기방향) 각도 사용.
- **검증(실제 클러스터링 재현)**: `구` 각도 -88°→0°, `북구`와 한 region 병합 확인. 세로글자(보내는분 3.2/밤는분 ~3/스탬프 OeVon 3.6)는 유지.

### 5-3. 주소 정규식 매칭 복구 — 쪼개진 토큰 재결합 (`OCR_pipeline_report.py build_full_text`)
- **문제**: 단어 분리가 한 토큰을 쪼개 공백을 넣어(`136길`→`13 6길`, `204호`→`204 호`) 주소 정규식 `[가-힣0-9]{1,10}(길|호)`가 공백을 못 넘어 매칭 실패. (격리 테스트: `136길`이면 매칭, `13 6길`이면 실패 확인)
- **해결**: `build_full_text`에서 한 줄 안 박스를 이을 때, **틈이 작고(`<글자높이×WORD_JOIN_GAP_RATIO`) + 앞이 숫자로 끝나며 + 뒤가 숫자/단위(호·동·층·번지·번·가·길·리)로 시작**하면 공백 없이 결합. 한글 단어(서울시+노원구)는 앞이 숫자가 아니라 안 붙음.
- 신규 상수: `WORD_JOIN_GAP_RATIO=0.8`, `WORD_JOIN_SUFFIX`.
- **검증(택배송장1)**: `204 호`→`204호` 결합 → 주소가 `서울시 노원구 동일로228길 청수빌딩 204호`까지 완전 탐지(이전엔 청수빌딩에서 끊김). `서울시 노원구` 공백 유지. 정규식·OCR박스·구역 로직 무수정.

### 5-4. 정규식 한계(정직 보고)
- 우편물3 상단 인쇄주소(대구 북구 송중대로)는 OCR 글자 깨짐(`여구 복 구칠 속중입대로`)이 심해 행정구역 토큰 소실 → 주소 탐지 실패. 손글씨 주소는 OCR이 `경북영덕`(군 누락)/`군영해면`(군 위치 이동)으로 읽어 부분 매칭. **인식 정확도 문제로, 정규식 보강은 오탐 위험이 커 미적용.**
