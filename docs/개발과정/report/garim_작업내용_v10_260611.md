# 작업 내용 v1 — 2026-06-11

## 주제: 상세보기(오버레이) 박스 — 같은 구역 단어별 다중 박스 → 단일 박스 통합 및 회귀 2건 해결

---

## 1. 배경 / 증상

같은 구역(zone)의 카드번호를 OCR 이 단어별 박스 여러 개("5678","9012","3455"…)로 잡아,
상세보기 영상에서 박스가 단어마다 제각각 표시되어 복잡했다.
→ "해당 구역에 가장 길게 잡힌 개인정보 박스 1개만" 표시되도록 요청.

수정 과정에서 **회귀 2건**이 발생했고, 최종적으로 근본 원인을 규명해 해결했다.

| 시점 | 증상 |
|------|------|
| 1차 수정 후 | 0:12(카드 최초 등장) 시점에 카드번호가 박스 없이 노출됨 |
| 2차 수정 후 | 박스가 위치·크기 모두 틀어짐(0:14 "3456"에만 작게, 0:12 엉뚱한 위치) |

---

## 2. 경로 구분 (중요)

| 경로 | 함수 | 역할 |
|------|------|------|
| 오버레이(상세보기 표시) | `_build_overlay_tracks` → `_render_overlay_video` | 화면에 박스 표시 |
| 마스킹(실제 지우기) | `_events_from_groups` → `_vid_track_template` | 픽셀 마스킹 |

→ 이번 수정은 **오버레이 경로에만** 적용. 마스킹 경로는 미변경.

---

## 3. 원인 진단 (코드 라인별 + 실측)

### 회귀 ① (1차 수정): 0:12 노출
- 1차 수정에서 `_pick_overlay_box`(면적 최대 박스 1개)를 **추적 입력**에 적용.
- 카드번호_2 의 최초 keyframe = **frame 394**. 이 keyframe 의 boxes 는
  큰 박스 없이 단어 3개("5678" 폭78, "9012" 폭81, "3455" 폭81)뿐.
- → 가장 작은 단어 1개(폭 81px)로 역추적 시작 → affine 매칭 시 추적 bbox 가
  좁아져 흐릿한 0:12 등장 구간에서 역추적이 일찍 끊김 → 박스 미도달(노출).

### 회귀 ② (2차 수정): 박스 위치·크기 틀어짐
- 2차 수정에서 추적은 전체 박스로 복원했으나, **표시**에 `_pick_overlay_box` 적용.
- frame 394 엔 큰 박스가 없어 단어 1개("3455")만 선택됨 → 그 1개 중심에
  전체 폭(`ref_w`)을 강제 적용 → 박스가 한쪽으로 쏠림.
- `_draw_box_on_frame` 은 quad 를 각각 그리므로 단어 1개 quad 만 표시됨.

### 공통 근본 원인
"면적 최대 박스 1개 선택"은 큰 박스가 있는 keyframe(frame 400)에서만 맞고,
단어만 있는 keyframe(frame 394)에서는 단어 1개로 축소되어 위치·범위가 깨진다.

---

## 4. 최종 해결

핵심: 단어 1개를 **고르는** 게 아니라, 추적된 단어들을 **하나로 병합**한다.

| 변경 | 위치 | 내용 |
|------|------|------|
| 추적 입력 복원 | `_build_overlay_tracks` evs 생성 | `_pick_overlay_box` 제거 → keyframe 전체 박스로 추적(역추적 안정성) |
| 표시 병합 | 표시 직전 | `_merge_boxes_to_quad` 신설 → 단어들 `cv2.minAreaRect` 로 회전 박스 1개 |
| 정리 | — | 미사용 `_pick_overlay_box`, `ref_w`/`ref_h`(오버레이) 제거 |

### `_merge_boxes_to_quad` 동작
- 모든 단어 박스 꼭짓점 → `cv2.minAreaRect` → `cv2.boxPoints` → 회전 quad 1개.
- 카드처럼 기울어진(angle) 개인정보도 전체를 딱 맞게 감싸는 단일 박스로 표시.

### 단위 검증 (frame 394 실측)
| | 박스 범위(x) | 결과 |
|---|---|---|
| 회귀(`_pick_overlay_box`) | 994~1075 ("3455"만) | 오른쪽 쏠림 ❌ |
| 최종(`_merge_boxes_to_quad`) | 811~1085 (전체) | 카드번호 전체 감쌈 ✅ |

병합 quad: `(811,353)→(1070,289)→(1085,350)→(826,414)` — 카드 기울기 반영.

## 5. 최종 동작

- 역추적: **전체 박스**로 진행 → 0:12 흐릿한 등장 구간까지 안정적 도달.
- 표시: 추적된 단어들을 감싸는 **회전 박스 1개** → 위치 정확 + 단일 박스로 깔끔.
- 추적·마스킹 경로 무변경.

## 6. 추가 회귀 — 박스 크기 들쭉날쭉 + 마스킹 미탐지 노출 (대표 박스 통일)

`_merge_boxes_to_quad` 적용 후에도 상세보기 박스 크기가 프레임마다 변하고(0:12 큼,
0:15 위치 틀어짐), 마스킹에서 frame 394 의 '1234' 미탐지 부위가 노출됨.

### 공통 원인
keyframe 마다 OCR 이 잡은 범위가 다름(frame 394=단어 3개, '1234' 누락 / frame 400=전체).
각 keyframe 을 따로 앵커로 쓰니 크기·범위가 제각각 + 미탐지 부위 누락.

### 해결: 그룹당 '대표 박스(전체가 가장 잘 잡힌 keyframe)' 단일 앵커
| 경로 | 변경 |
|------|------|
| 오버레이 `_build_overlay_tracks` | 대표 keyframe(병합 면적 최대=frame 400) 1개를 단일 앵커로 전 구간 추적. 표시 박스는 크기·각도 고정(353×40, -13.5°), 위치만 이동 → 크기 일정 + 회전 + 미탐지 부위 포함 |
| 마스킹 `_vid_track_template` | 대표 keyframe 단어별 박스(컨테이너 제외)를 단일 앵커로 추적 마스킹 → 타이트 + '1234' 포함 |
| 신규 헬퍼 | `_drop_container_boxes`: 단어 2개+ 감싸는 전체 묶음 박스 제외(타이트), 단어별 없으면 원본 유지(노출 방지) |

### 사용자 의도 반영
"기존 탐지 안 된 부위를 탐지·제거"가 최대 목적 → 대표 박스(전체 단어 커버)로
모든 프레임에서 미탐지 부위까지 제거. 하이브리드(다중 앵커 위치 보정)는 표류 확인 후 추가.

### 단위 검증
- 대표 선정: frame 400(병합면적 14223 최대) ✅
- 크기 일정: 위치 이동해도 353×40 유지 ✅
- 컨테이너 제외: 큰 박스(725~1077, 단어4개 포함) DROP, 단어4개 KEEP ✅

## 7. 라벨 위치·각도 개선 + 마스킹 깜빡임 해결 (다중 완전 앵커 = 하이브리드 완성)

### 작업 1 — 라벨(제목)이 박스 안 내용과 겹침
- 증상: 'Card-Number2' 라벨이 박스 안에 수평으로 그려져 카드번호와 겹쳐 둘 다 안 보임.
- 해결(신규 헬퍼):
  - `_quad_label_angle`: 박스 긴 변 각도 → -45~45 정규화(거꾸로/세로 뒤집힘 방지).
  - `_put_rotated_label`: 박스 각도로 회전한 라벨을 박스 '밖'(위 우선, 위 짤리면 아래)에
    배치, 화면 경계 클램프, 검은 외곽선으로 가독성 확보.
  - `_draw_box_on_frame`: 위 헬퍼로 라벨 그리도록 교체(quad/bbox 양쪽).
- 검증: 카드 각도 -13.9° 반영, 최상단 박스는 라벨이 아래로 배치(짤림 방지) ✅

### 작업 2 — 결과영상 마스킹 깜빡임(지워졌다 살짝 보였다 반복)
- 원인: 직전 단계의 '단일 대표 앵커(frame 400)'가 멀어진 구간에서 누적 추적오차 →
  마스킹 박스가 카드 미세 움직임을 못 따라가 가장자리 노출 → 다음 프레임 또 잡힘(깜빡임).
- 해결: 다중 '완전' 앵커(`VID_FULL_ANCHOR_RATIO=0.8`)
  - 단어 병합면적이 그룹 최대의 80% 이상인 완전 keyframe 만 앵커로 사용.
  - 마스킹(`_vid_track_template`)·오버레이(`_build_overlay_tracks`) 양쪽 적용.
  - 각 완전 keyframe 이 '자기 실측 위치·각도'로 인접 구간만 추적 → 누적오차 주기적 리셋
    → 카드 미세 움직임을 정확히 추종(삐짐/깜빡임 방지).
  - 일부만 잡힌 keyframe(frame 394/469)은 앵커에서 제외, 인접 완전 앵커의 추적이
    '1234' 포함해 커버 → 미탐지 노출 방지.
  - 오버레이 표시는 여전히 '대표 크기·각도 고정'(크기 일정).

### 완전 앵커 선별 검증
- 완전 앵커: frame 400, 424, 430, 448, 478 (5개)
- 제외: frame 394, 469 (일부 단어만 — '1234' 등 누락)

## 8. 라벨 좌상단 배치 + 회전(affine) 추적 강화

### 작업 A — 라벨이 박스에서 너무 위로 떨어짐
- 변경: `_put_rotated_label` 위치 로직을 '박스 중심 위' → '박스 좌상단 바로 위(좌측 정렬)'.
  - 위가 잘리면 박스 바로 아래, 우측이 잘리면 우측 정렬로 당김 → 안 가리고 안 잘림.
- 검증: 박스 좌상단(x≈811, top≈289) → 라벨 x[816~] y[227~276] 좌상단 바로 위 ✅

### 작업 B — 0:15 카드 회전 시 끝글자('3456') 노출
- 원인: `_vid_match_shift` 가 전체 NCC 매칭만 thresh 넘으면 무조건 'shift'(평행이동) 반환 →
  카드가 회전해도 박스는 안 돌아 끝자리가 박스 밖으로 나가 노출.
- 해결: 블록 affine 을 항상 먼저 추정하고, 한 프레임 회전(`VID_AFFINE_PREFER_ROT=1.0°`) 또는
  스케일변화(`VID_AFFINE_PREFER_SCALE=0.02`)가 유의미하면 shift 대신 affine 우선 채택.
  - 회전이 작으면 기존처럼 전체 shift(안정), 클 때만 affine(회전·원근 추종).
  - `_vid_affine_sane`(스케일 0.6~1.7, 각도 ≤25°)이 폭주 방지.
- 효과: 카드 회전·원근 변화 시 마스킹/표시 박스가 같이 회전 → 끝글자까지 추종.

## 9. 영상 회전 메타데이터 보정 (별개 이슈 — 다른 테스트 영상)

### 증상
새 테스트 영상(음성_영상1)에서 상세보기·결과 영상이 90도 누워서 출력.

### 진단 (해상도 실측)
| 단계 | cv2 해상도 | 비고 |
|------|-----------|------|
| OCR(index.json) | 720×1280 세로 | ffmpeg 백엔드가 회전메타 적용 |
| 상세보기·결과 | 1280×720 가로 | 로컬 cv2 가 회전메타 무시(raw) |
- 원인: 폰 세로영상(시계 90° 메타). OCR 은 세로로 처리(좌표도 720×1280 기준)했으나
  로컬 렌더링 cv2 가 메타를 무시하고 가로 raw 를 읽어 90도 회전 출력.
- 라벨/추적 로직과 무관한 '영상 입력 방향 처리 누락'.

### 해결 (공통 헬퍼 자동 보정)
- 전역 `_FRAME_ROT` + `_set_frame_rot(cap, ref_w, ref_h)` + `_rot_read(cap)` 신설.
  - run_* 진입 시 cap 해상도가 index(OCR) 기준과 90도 어긋나면 회전코드 설정.
  - 메타(`CAP_PROP_ORIENTATION_META`) 우선, 없으면 폰 세로=시계방향 기본.
  - 모든 프레임 읽기(`cap.read()` 8곳 → `_rot_read`)에서 동일 방향 보정.
  - 진입점(run_detail_view / preview_video / mask_video_full)에서 `_set_frame_rot` 호출 +
    출력 W,H 스왑 → VideoWriter·트랙·마스킹 모두 세로 좌표계로 일관.
- 양 환경 안전: cv2 가 이미 메타 적용(720×1280)하면 '일치'로 판단해 회전 안 함.

### 단위 검증
- index 720×1280 vs cap 1280×720 → ROTATE_90_CLOCKWISE ✅
- 회전 후 프레임 (1280,720,3)=720×1280 세로 ✅
- 이미 세로면 None(회전 안 함) ✅

## 10-A. 주소 정규식 보강 (OCR report)

### 증상 (STT json 음성_영상1 기준 — 동일 패턴이 OCR 에도 적용)
주소 3개 중 2개 누락:
| 주소 | 기존 | 비고 |
|------|------|------|
| 대전시 유성구 대학로 56 | 탐지 | 구 + 한글로 + 숫자 |
| 울산시 남구 33로 78 | **누락** | 숫자로 시작 도로명(33로) |
| 세종시 도음 3로 90 | **누락** | 구/동 없는 중간어(도음) + 숫자로(3로) |

### 원인
기존 주소 패턴(분기1)이 ① 중간어가 시/군/구/동으로 끝나야 하고('도음' 불가)
② '로' 앞이 한글 위주라 '33로/3로'(숫자로 시작 도로명)를 놓침.

### 해결 (OCR_pipeline_report.py — 기존 무수정, 14번 패턴 추가)
사용자 규칙 기반 새 주소 분기 추가:
```
[가-힣]{2,3}시 [공백] (중간 1~2단어, 6글자 이내, 구/동 무관){1,2} [공백] [가-힣0-9]{1,10}로 [공백] 숫자
```
- 'OO시'로 시작 + 중간(구/동 또는 6글자 임의어) + 'XX로'(숫자로 허용) + 번지 숫자(필수).
- 기존 패턴은 그대로 두고 OR 보강 → 탐지 로직 `is_overlap`(331~334)이 중복 자동 제거.

### 검증
- 누락 2개 포함 4개 정탐, 오탐('시내버스 타고 어디로 3') 없음 ✅
- py_compile 통과 ✅

### STT(colab_pipeline_stt.py) — 보강 완료
STT 도 같은 주소 패턴이라 동일 누락. STT `_detect_regex`(504~)에는 중복 제거가 없어,
별도 패턴 추가(중복 위험) 대신 **기존 12번 주소 패턴 안에 OR 로 새 분기를 합침**(패턴 1개
유지 → 중복 segment 방지). 검증: 전체 STT 텍스트에서 주소 3개(대전/울산/세종) 모두 탐지,
py_compile 통과.

### OCR vs STT 적용 방식 차이 (의도적)
| | 중복 제거 | 적용 방식 |
|---|---------|----------|
| OCR (is_overlap 있음) | 자동 | 14번 '별도 패턴' 추가 |
| STT (중복 제거 없음) | 없음 | 12번 패턴에 'OR 내장' (1회 매칭) |
