## 1. FFmpeg h264_nvenc GPU 인코딩 교체 (전 세션 완료분)

### 병목
- `cv2.VideoWriter(mp4v)` = CPU 인코딩. 900프레임 전체를 CPU로 인코딩 → 수 분 소요.
- LaMa 인페인팅은 이미 PyTorch CUDA 자동 사용 중(병목 아님).

### 수정 (colab_pipeline_mask.py, 3군데)
1. `_check_nvenc()` 헬퍼 추가 — FFmpeg `h264_nvenc` 지원 여부 1회 캐싱 확인.
2. `_video_mask_to_temp()` — `cv2.VideoWriter` → `subprocess.Popen(ffmpeg, stdin pipe)`.
   프레임을 `.tobytes()`로 FFmpeg에 직접 스트리밍, `-c:v h264_nvenc`(GPU) 인코딩.
3. `_render_overlay_video()` — 상세보기 오버레이 영상도 동일 GPU 파이프 적용.

### 폴백
- `USE_GPU=False` 또는 `h264_nvenc` 미지원 시 자동으로 `libx264`(CPU) 사용.
  → 기존 동작과 완전 호환.

### 효과 (Colab T4 기준)
- 영상 인코딩 3~5배↑.

## 3. ★ 오버레이 트랙 생성(역방향 추적) 디코딩 병목 제거 (금일 핵심)

### 사용자 보고
- "오버레이 트랙 생성" 단계가 너무 오래 걸림.

### 병목 진단
- 진짜 병목은 GPU로 풀 수 있는 연산(`cv2.matchTemplate`)이 **아니라** 영상 디코딩 방식.
- `_track_one_dir`, `_vid_track_run`의 **역방향(등장 역추적) 프레임별 random seek**:
  ```python
  cap.set(cv2.CAP_PROP_POS_FRAMES, nfi)   # 매 프레임 seek
  ret, cur = _rot_read(cap)
  ```
  H.264는 임의 프레임 seek 시 직전 키프레임부터 재디코딩 → 한 칸 뒤로 가는 데
  수십 프레임 디코딩(사실상 O(n²)). 이게 전체 시간의 대부분.
- `cv2.cuda.matchTemplate`(GPU)는 Colab 기본 opencv-python이 CUDA 빌드가 아니라 사용 불가.
  → 병목이 디코딩 구조이므로 GPU 없이 구조 개선만으로 해결.

### 수정 내용 (블록 prefetch)
- 역방향을 **블록(64프레임) 단위 1회 seek + 순차 디코딩 → 캐시에서 역순 pop** 방식으로 교체.
- `_track_one_dir`(상세보기 트랙용), `_vid_track_run`(실제 마스킹 추적용) **두 함수 모두** 적용.
- 함수 내부 클로저 `_read_cur(target_fi)` 헬퍼:
  - 정방향: 기존 `_rot_read(cap)` 순차 디코딩(변경 없음).
  - 역방향: 캐시에 없으면 `[block_lo, target_fi]`를 한 번 순차 디코딩해 채우고 pop.
    seek 횟수가 1/64로 감소.

### ★ 추적 정확도 보존 (사용자 핵심 우려)
- 사용자 요건: "뒤에서 3단어 다 잡은 대표 keyframe으로 앞쪽 흐린 구간을 역추적해서,
  약간 흐려도 동일 픽셀이면 3단어 전체 범위 박스를 잡아준다" — 이 동작이 깨지면 안 됨.
- 보존 근거: prefetch는 **같은 프레임 인덱스를 디코딩**하므로 `_rot_read()` 결과 픽셀이
  seek 방식과 **비트 단위로 동일**. 따라서 `matchTemplate` 결과, `corr_ok`(_patch_corr) 판정,
  대표 박스(rep_size·rep_ang 고정) 좌표가 **전혀 바뀌지 않음**.
- 관련 코드: 대표 앵커 선택 `_build_overlay_tracks` rep_kf(면적 최대 keyframe),
  동일 픽셀 판정 `corr_ok = _patch_corr(...) >= VID_REF_CORR`.

### 검증
- AST 파싱 OK.
- `CAP_PROP_POS_FRAMES, nfi` per-frame seek 잔존 0건 확인.
- 메모리 상한: 720x1280 BGR × 64프레임 ≈ 170MB (Colab RAM 안전 범위).

## 4. ★ 다량 PII 밀집 환경 — drift lock (오매칭·표류 방지) 추가 (금일)

### 사용자 보고
- 종이에 전화번호 7개+주소 13개가 빽빽이 배열된 영상에서 박스가 인근 PII로 점프하거나 사라지는 현상.
- "Phone4 박스가 중간에 사라지고 Phone5가 나타남 → 조금 뒤 Phone5도 사라지고 Phone6 나타남"
- "일부 주소 박스가 갑자기 위치 이탈"

### 원인 진단
1. **VID_SEARCH_RATIO=0.8 탐색 범위가 인근 전화번호를 포함** (전화번호 간격 ~30px < 탐색 범위 148px)
2. **_patch_corr 검증 미흡** — 동일 폰트·포맷의 전화번호 패치가 서로 유사해 표류 후에도 corr 0.40 통과
3. **VID_REF_CORR_STREAK=2** — 2프레임 연속 실패 시 즉시 박스 제거 → Phone4 사라짐 → Phone5 독립 추적 시작

### 수정 내용 (drift lock 메커니즘)
- 파라미터 추가 (`VID_DRIFT_LOCK_RATIO=1.5`, `VID_DRIFT_MIN_PX=20`, `VID_DRIFT_RETRY_RATIO=0.15`)
- `_track_one_dir`(상세보기)·`_vid_track_run`(마스킹) 양 함수에 동일 적용
- 로직: 추적 박스 중심이 앵커 원점에서 **박스 높이×1.5** 이상 이탈 시 오매칭으로 판단
  → 원점 근방 좁은 범위(0.15)에서 재탐색 → 성공: 교정 위치 / 실패: 이전 위치 유지(사라짐 없음)
- `_build_overlay_tracks`, `_vid_track_template` 호출부에 `anchor_bbox=` 전달

### 속도 영향
- 정상 케이스(대부분): float 연산 몇 개만 추가 → **무시 가능**
- 이탈 케이스(오매칭 시): `_vid_ncc_probe` 재탐색 1회 추가, 탐색 범위 0.15(기존 0.8보다 빠름)
- **속도 저하 없음**

### AST 검증
- AST 파싱 OK
- drift lock 블록 3개, anchor_bbox= 호출 6개, VID_DRIFT_RETRY_RATIO 참조 3개 확인
