# 작업 내용 v1 — 2026-06-10

## 주제: 오디오 마스킹 병합 후 발생한 카드번호 탐지/추적 이상 4건 해결

---

## 1. 배경 / 증상

오디오 마스킹 기능을 병합한 뒤 영상 카드번호 처리에서 4가지 이상이 보고됨.
사용자 확인 결과 **"기존(검증테스트) 코드로 돌려도 동일"** → 병합 코드 버그가 아니라
원래부터 존재하던 알고리즘 특성 + 누락이 원인.

| # | 증상 |
|---|------|
| ① | 첫 번째 카드의 카드번호 박스를 못 잡음 |
| ② | 두 번째 카드에서 범위 박스가 2개씩 잡힘 |
| ③ | 카드 등장 전에 박스가 화면 위에서 내려옴 |
| ④ | 오디오 마스킹 "삐(비프)" 소리가 안 남 |

---

## 2. 원인 진단 (코드 라인별 비교 + 디버깅 파일/JSON 실측)

- 검증테스트 ↔ 현재 폴더의 **mask 영상처리 코드는 완전 동일**(diff 확인).
- report 코드 차이는 전화번호 패턴 1곳뿐, `OCR_Total.py`는 업스케일 배율 2개뿐.
- `_index.json` / `_result.json` 도 두 폴더 **바이트 단위 동일** → 입력 데이터도 같음.
- 즉 코드 차이가 아니라 **알고리즘 동작 특성 + 기능 누락**이 원인이었음.

| # | 진짜 원인 | 위치 |
|---|----------|------|
| ① | 카드번호 정규식이 **16자리(4-4-4-4) 고정** → OCR이 끝 4자리를 놓친 12자리(`1234 5678 9876`)는 탈락. (OCR 텍스트엔 분명히 존재) | `OCR_pipeline_report.py` 카드번호 패턴 |
| ② | `_vid_cluster_events`가 클러스터 대표 key를 **"가장 긴 텍스트"로 갱신** → OCR 박스 중복인식으로 비정상적으로 길어진 변형이 대표가 되어, 정상 프레임(유사도 0.938)과의 비교가 0.7 미만으로 떨어져 같은 카드가 2그룹 분리 | `OCR_pipeline_report.py` `_vid_cluster_events` |
| ③ | 상세보기 트랙 생성 시 첫 keyframe **이전 2초**(`VID_EDGE_MAX_SEC`)까지 역방향 NCC 추적 → 카드 등장 전 같은 장면 구간에서 배경을 오매칭해 박스가 위에서 표류 | `colab_pipeline_mask.py` `_build_overlay_tracks` |
| ④ | `_generate_beeped_audio()`가 **정의만 되고 호출 안 됨** + 필요 상수 `BEEP_FREQ`/`BEEP_GAIN_DB` **미정의**(호출 시 NameError) + `_mux_audio`는 항상 원본 오디오만 mux | `colab_pipeline_mask.py` |

## 3. 수정 내용

### ④ 비프음 (mask)
- `BEEP_FREQ=1000.0`, `BEEP_GAIN_DB=-3.0` 상수 신규 정의.
- `_mux_audio(..., audio_path=None)` 인자 추가 → 비프 wav가 있으면 그것을 오디오 소스로 사용.
- `mask_video_full`에서 `_generate_beeped_audio()` 호출 → 비프 wav 생성 → mux 연결 → 임시 wav 정리.
- 오디오 PII가 없거나 미선택이면 기존대로 원본 오디오 사용(동작 보존).

### ③ 위에서 내려옴 (mask, 표시 전용)
- `VID_ENTER_MAX_SEC=0.3` 상수 신규 정의(첫 keyframe '이전' 등장 전 역추적 한계).
- `_build_overlay_tracks`의 등장 전 역추적을 `edge_max`(2초) → `enter_max`(0.3초)로 단축.
- **마스킹 함수(`_vid_track_template`)는 미변경** → 개인정보 가림 범위는 그대로(보안 우선).

### ② 박스 2개 분리 (report)
- `_vid_cluster_events`: 대표 key 1개 비교 → **클러스터 멤버 전체 텍스트 중 최대 유사도**로 판정.
- 같은 카드의 OCR 변동은 흡수, 진짜 다른 카드는 여전히 분리.

### ① 첫 카드 미탐지 (report)
- 카드번호 정규식을 **구분자 그룹화 4-4-4 / 4-4-4-4 + 붙여쓴 15~16자리**로 확장.
- 12자리 '연속'은 오탐 위험으로 제외(구분자로 그룹화된 4-4-4만 카드 인정).
- 신/구 정규식 테스트: 첫 카드 포착 ✅, 오탐 샘플 8종(날짜·전화·계좌·금액·우편) 전부 오탐 0 ✅.

## 4. 검증을 위한 재실행 순서 (중요)

①②는 **report 단계 수정**이라 JSON을 다시 만들어야 반영됨:

1. `OCR_pipeline_report.py` 재실행 → `{stem}_index.json` 재생성 (①첫 카드 탐지, ②박스 1개)
2. `backend_json_merger.py` 재실행 → `{stem}_result.json` 재생성 (오디오 그룹 병합)
3. `colab_pipeline_mask.py` 재실행
   - `run_detail_view` → `_상세보기.mp4` (③박스 표류 해소 확인)
   - `run_masking` → `_결과.mp4` (④비프음 확인, `pydub`·`ffmpeg` 필요)

> ※ OCR 원본(`ocr_index.json`/text)은 그대로 재사용 가능 — 정규식/그룹화만 바뀌었으므로 OCR_Total 재실행 불필요.
> ※ 참고: `OCR_Total.py` 업스케일 배율이 검증테스트(2.0/3.0)보다 낮음(1.7/2.5). 첫 카드는 정규식으로 해결됐으나, OCR 정밀도를 더 높이려면 배율 원복도 선택지(미적용).

## 10. 다른 장면에 박스 잔존 — 사라짐 '연속 판정' 도입 (동일 일자 업데이트)

### 데이터 실측 진단 (카드_음성_영상1)
- `output_file/카드_음성_영상1_result.json` 그룹 구성 실측:
  - `카드번호_1`: keyframe=`[0]` 1개, center(687,361) — frame 0 의 `1234 5678 9876 5432`.
  - `카드번호_2`: keyframe=`[394,400,424,430,448,469,478]`, center(900~948,350) — 13~16초 카드2.
- **0:00·0:16 박스는 표류가 아니라 실제 OCR 데이터**였음(frame 0=카드1, frame 478=카드2 재등장).
  keyframe 은 OCR 확정값이라 픽셀 비교 없이 표시되는 것이 정상.
- **진짜 표류 지점**: 카드번호_2 의 첫 keyframe(394=13초)에서 frame 0(0초)까지 역추적하는 구간.
  카드2 가 13초 전엔 화면에 없는데, 사라짐 판정(`_patch_corr < VID_REF_CORR=0.30`)이 너무 관대해
  다른 장면(검은화면·말풍선·다른 컷)을 '아직 카드'로 오인 → 박스가 0초 쪽으로 표류
  ("왼쪽에서 날아옴 / 0:00 우상단 / 말풍선 박스"의 진짜 원인).

### 핵심 인식
- 추적은 **이미 매 프레임(±1) NCC 비교**를 수행 중(="미세 프레임 미검사"는 사실 아님).
- 빠진 것은 **"원본 PII 와 픽셀이 확연히 달라지면 = 사라진 것 = 박스 제거"** 판정의 엄격함.
  기존 `VID_REF_CORR=0.30` 단일 프레임 판정은 거의 끊기지 않았음.

### 수정 (mask, 표시·마스킹 양쪽 동일)
- `VID_REF_CORR` 0.30 → **0.40** 상향.
- `VID_REF_CORR_STREAK = 3` 신규: 상관도 미달이 **연속 3프레임** 지속될 때만 추적 중단.
  단일 프레임 순간 흐림/노이즈는 견뎌 흐릿한 등장 보존, 실제 장면 전환만 박스 제거.
- `_track_one_dir`(표시)·`_vid_track_run`(마스킹)에 `miss_streak` 카운터 추가
  (미달 누적/유사 복귀 시 초기화). 기존 NCC·외삽·순간이동·가까운앵커 로직은 불변.
- 임계 2개(`VID_REF_CORR`, `VID_REF_CORR_STREAK`)는 상수로 노출 → 영상 보며 사용자 조정.
- **특정 영상 하드코딩 없음** — 박스 크기 비례 순간이동 + 패치 48×48 정규화로 모든 영상/PII 범용.
- `py_compile` 통과.

## 11. 검은화면에 표시 박스 잔존 — 렌더 hold/grace 기록 정리 (동일 일자 업데이트)

### 증상
- 10절(연속 판정) 적용 후에도 **장면 전환으로 텍스트가 완전히 사라진 검은 화면에 범위 박스가 남음**.
- 이는 **마스킹과 무관한 '상세보기 표시 박스'** 문제(개인정보 위치 표시 오버레이).

### 원인 진단 (코드 실측, 표시 경로 한정)
- 추적(`_track_one_dir`)은 매 프레임 NCC 비교로 장면 전환 지점에서 **정상적으로 끊김**.
- 그런데 박스가 남는 이유는 표시 경로 두 곳이 겹쳤기 때문:
  1. **렌더러 hold 과대** — `_render_overlay_video` 가 마지막 추적점을 `step*2` 프레임 유지.
     정상 추적 구간 점 간격은 `step` 인데 hold 가 2배(`step*2`)라, 트랙이 끝나는 지점(=소멸)에서도
     마지막 박스가 `step*2`(예 30fps·TRACK_FPS10 → 6프레임 ≈0.2초) 동안 검은 화면으로 끌려감.
  2. **grace 프레임 기록** — `miss_streak` 연속 판정이 끊기기 전(grace, 최대 STREAK-1=2프레임)
     동안 유사도 미달 프레임의 박스를 `_rec` 로 그대로 기록 → 검은 화면 점이 트랙에 남음.

## 12. 오버레이 박스 다중 표시 → 1개 단일 박스로 통합 (2026-06-11)

### 증상
같은 카드번호 구역에서 `"1234"`, `"5678"`, `"9012"`, `"3455"` 같은 단어별 개별 박스가
화면에 여러 개 동시에 표시되어 복잡해 보임.  
(예: 0:12 시점에 작은 박스 여러 개 → 0:13 시점에 큰 박스 1개 — 프레임마다 불일치)

### 원인
`카드번호_2` pii_group의 `keyframes[x]['boxes']` 배열에 **전체 카드번호 박스(큰 것) +
개별 단어 박스들**이 모두 포함되어 있음.
`_build_overlay_tracks`에서 이 모든 boxes를 그대로 추적에 사용하고,
`_bbox_quad_from_boxes` → `_draw_box_on_frame`에서 quads를 개별로 모두 그림.

### 수정 내용 (`colab_pipeline_mask.py`, `_build_overlay_tracks` 함수)
```python
# 추가: 헬퍼 — 면적 최대 박스 1개만 선택
def _pick_overlay_box(boxes):
    if not boxes:
        return boxes
    return [max(boxes, key=lambda b: (b.get('x_max', 0) - b.get('x_min', 0)) *
                                      (b.get('y_max', 0) - b.get('y_min', 0)))]

# 변경 전
evs = sorted([(kf['frame'], _expand_boxes(kf['boxes'], VID_TRACK_EXPAND), ...)
# 변경 후
evs = sorted([(kf['frame'], _expand_boxes(_pick_overlay_box(kf['boxes']), VID_TRACK_EXPAND), ...)
```

### 효과
- 각 keyframe에서 **면적이 가장 큰 박스 1개만** 추적 앵커로 사용
- 오버레이에 개인정보 박스 1개만 표시 — 단어 박스 난립 해소
- 실제 마스킹 경로(`_vid_track_run`)는 미변경 — 가림 정확도 유지
- 픽셀 NCC 추적도 가장 큰 박스 기준으로 통일 → 추적 안정성 향상
- 모든 영상/PII 범용 (특정 영상 하드코딩 없음)
