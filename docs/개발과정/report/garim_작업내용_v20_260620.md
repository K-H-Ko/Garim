## 📌 세션 1: 결과 페이지 구간 다운로드 기능 구현

**배경**: `ocr_mask/(참조)남은과제.md` 3번 항목 — "결과 페이지 구간 다운로드 기능 미구현(프론트페이지에서 구현)"

기존 ResultPage.jsx는 처리 완료된 파일을 전체 다운로드만 지원. 순수 브라우저 API(Canvas, MediaRecorder)를 활용하여 백엔드 변경 없이 구현.

### 구현 내용

#### 파일 타입별 분기

| 파일 타입 | 구간 다운로드 방식 |
|----------|-----------------|
| 이미지 | Canvas 오버레이 + 마우스 드래그 영역 선택 → PNG 크롭 다운로드 |
| 영상 | 시작/종료 시간(mm:ss) 입력 → MediaRecorder + captureStream() → WebM 다운로드 |

#### 이미지 크롭 동작 흐름
1. "구간 다운로드" 토글 클릭 → 이미지 위에 `<canvas>` 오버레이 활성화 (cursor: crosshair)
2. 마우스 드래그 → 어두운 배경 + 파란 점선 박스 + 모서리 핸들 실시간 렌더링
3. "영역 다운로드" 버튼 → `fetch(fileUrl, { credentials: "include" })` 로 이미지 Blob 획득
4. offscreen Canvas `drawImage(srcImg, x, y, w, h)` → `toBlob("image/png")` → PNG 다운로드
5. canvas 좌표 = 실제 이미지 자연 해상도 좌표 (자동 스케일 변환)

#### 영상 구간 동작 흐름
1. 시간 입력 (mm:ss 형식) + 유효성 검사 (start < end, end ≤ duration)
2. `video.currentTime = startSec` → `seeked` 이벤트 대기 (탐색 완료 보장)
3. `video.captureStream()` + `MediaRecorder` 시작 (vp9 → vp8 → webm 순 codec 선택)
4. `timeupdate` 이벤트로 endSec 도달 감지 → `recorder.stop()`
5. `recorder.onstop` → Blob 합산 → `.webm` 다운로드
6. 녹화 중 "● 녹화 중..." 배지 + "취소" 버튼 제공

### 추가된 UI 구성

```

│       └── 선택 완료 힌트
```

#### 신규 파일: `backend/services/file_cleanup.py`

- `cleanup_expired_files()`: `expires_at <= now() AND deleted_at IS NULL` 인 `processed_files`를 최대 200건씩 조회
  1. 실제 파일 삭제 (`os.remove`)
  2. `processed_files.deleted_at = now()` 소프트 삭제 표시

- `start_cleanup_scheduler()`: 데몬 스레드 시작 → 서버 기동 후 60초 대기 → 이후 1시간마다 실행

- `lifespan()` 함수에 `start_cleanup_scheduler()` 호출 추가 — 서버 시작 시 스케줄러 자동 기동

## 📌 세션 4: deletion_events 자동삭제·사용자 임의삭제 구분

**배경**: 기존 `deletion_events` 기록은 만료 자동삭제(스케줄러)만 저장됨. 히스토리 페이지에서 사용자가 직접 삭제 버튼을 눌러도 이력이 남지 않아, 자동삭제와 임의삭제를 구분할 수 없는 문제.

### 구분 기준

| 컬럼 | 자동삭제(만료) | 사용자 임의삭제 |
|------|--------------|--------------|
| `actor_type` | `'system'` | `'user'` |
| `delete_reason` | `'expired'` | `'user_request'` |
| `actor_user_id` | `NULL` | 삭제 요청한 사용자 UUID |
| `scheduled_delete_at` | 만료 예정 시각 | 원래 만료 예정 시각(참고용) |

### 수정 내용: `backend/services/analysis.py` — `delete_upload_and_jobs()`

1. **사전 조회 추가** — `processed_files` DELETE 전에 해당 파일들의 `processed_file_id`, `stored_path`, `expires_at`을 `pf_records`로 수집 (DELETE 후 조회 불가)
2. **deletion_events INSERT 추가** — 모든 DB 삭제 완료 후, `pf_records`를 순회하며 `actor_type='user'`, `delete_reason='user_request'`, `actor_user_id=user_id`로 이력 기록
3. **이력 기록 실패 격리** — INSERT 실패 시 경고 로그만 남기고 삭제 자체는 계속 진행 (UX 저해 방지)

### 흐름 요약

```
히스토리 페이지 [삭제] / [전체 삭제] 클릭
    → DELETE /analysis/uploads/{upload_id}
    → delete_upload_and_jobs(upload_id, user_id)
        ① processed_files 정보 사전 조회 (pf_records)
        ② 파일·DB 레코드 삭제
        ③ deletion_events INSERT (actor_type='user', delete_reason='user_request')
        ④ db.commit()
```

### 수정 파일

| 파일 | 변경 |
|------|------|
| `backend/services/analysis.py` | `delete_upload_and_jobs()` — pf_records 사전 조회 + deletion_events 사용자 삭제 이력 INSERT |

4개 탭 모두 실제 DB와 연결하고, 탭 전환을 React state 기반으로 재구성.

### 신규 API (4개)
- `GET /admin/compliance/overview` — 자동삭제 탭 (정책 현황 + 삭제 예정 + 최근 로그)
- `GET /admin/compliance/search?q=...&type=job_id|user_id|watermark` — 처리 이력 검색
- `GET /admin/compliance/consent?q=...` — 약관 동의 이력 (email 또는 user_id)
- `GET /admin/compliance/reports` — 신고·수사 응답 이력

### 백엔드 변경

| 파일 | 변경 |
|------|------|
| `backend/services/admin.py` | `get_compliance_overview()`, `search_compliance()`, `_build_job_result()`, `get_consent_history()`, `get_compliance_reports()` 추가 |
| `backend/controllers/admin.py` | 4개 핸들러 추가 |
| `backend/routes/admin.py` | 4개 라우트 등록 |

#### 주요 쿼리 테이블
- `uploads`, `processed_files` → 원본/결과 파일 잔존 수 및 준수율
- `analysis_jobs`, `users` → 메타데이터 건수, 탈퇴 회원 유예 수
- `deletion_events` → 최근 24h 자동 삭제 로그
- `user_consents` → 약관 동의 이력
- `abuse_reports` → 신고·수사 응답 이력

### 프론트엔드 변경

| 파일 | 변경 |
|------|------|
| `frontend/src/utils/api.js` | 4개 API 함수 추가 |
| `frontend/src/pages/garim/AdminCompliance.jsx` | 전면 교체 — React state 탭 전환 + 실제 DB 연동 |
| `frontend/src/css/garim-pages/AdminCompliance.css` | `compliance-pill.err` 추가 |

#### AdminCompliance.jsx 주요 특징
- 탭 전환: `data-tab` 속성 + JS 대신 `activeTab` state + 조건부 렌더링으로 재구성
- auto 탭: 10분 폴링 (실시간성 낮아도 OK)
- search/consent 탭: 탭 진입 시 기본 최근 50건 자동 로드 + 입력 후 버튼 클릭 시 검색
- report 탭: activeTab === "report" 최초 진입 시 1회 로드
- 준수율 계산: `(1 - overdue/total) * 100` — expires_at 초과 미삭제 건 기준

### 주요 변경점

#### 1. Nginx 및 Docker Compose 분리/연동
