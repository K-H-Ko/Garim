### 2-2. OCR "청수빌딩 204호" 미탐지 원인 분석 및 해결

**원인 파악:**

OCR JSON 분석 결과, OCR은 "청수빌딩 204호"를 정확히 인식하고 있었음.

문제는 `_split_box_by_words()` 적용 순서에 있었음:

**기존 파이프라인 (잘못됨):**
```
load_ocr_boxes() → _split_box_by_words() → universal_directional_clustering() → full_text
```

`_split_box_by_words()`로 박스가 단어별로 분리되면, 분리된 박스들의 proj_y 값이 달라지고 클러스터링 순서가 바뀌어 "착불소"(x=395~457)가 "서울시노원구동일로228길35" 줄에 잘못 끼어들었음:

```
[잘못된 결과] 줄 14: ['주소', '서울시노원구동일로228길35', '착불소']
full_text: "...서울시노원구동일로228길35 착불소  청수빌딩 204호..."
→ 패턴 14번이 '착불소' 앞에서 끊겨 청수빌딩 204호 미포함
```

**OCR_Setting.py 동작 방식 분석:**

OCR_Setting.py의 `assemble_text()`는 원본 박스를 기준으로 클러스터링하므로:
- "010.4201.7478"이 먼저 줄을 구성
- "착불소"가 "010.4201.7478" 줄에 추가됨 (X겹침 없음, proj_y 근접)
- "서울시..."가 "010.4201.7478"과 X축 142픽셀 겹침 → 다른 줄로 분리

결과: "착불소 안심(보험)" 줄과 "주소 서울시노원구동일로228길35" 줄이 분리됨

**수정 방향 (OCR과 동일한 방식으로 join):**

`_split_box_by_words()`를 클러스터링 전이 아니라 PII 박스 수집 후에 적용하도록 순서 변경:

```python
# 수정 후 파이프라인
load_ocr_boxes()           # 원본 박스 (단어 분리 없음)
→ universal_directional_clustering()  # 원본 박스 기준 클러스터링 (OCR과 동일)
→ build_full_text()        # 올바른 full_text
→ analyze_and_mask_text_all()  # PII 탐지
→ _split_box_by_words()   # PII 박스에만 단어 분리 적용 (마스킹 정교화)
```

**수정된 파일: `ocr_mask/OCR_pipeline_report.py`**

1. `load_ocr_boxes()`: `_split_box_by_words()` 호출 제거 (원본 박스만 반환)
2. `analyze_image_one()` 내 PII 박스 수집 후: `_split_box_by_words()` 적용 (마스킹 정교화)

**검증 결과:**

```
클러스터링 결과 (수정 후):
  줄 11 (r=1): ['보내는분', '010.4201.7478']   ← 착불소 제거됨 ✓
  줄 12 (r=1): ['착불소', '안심(보험)']          ← 착불소 분리 ✓
  줄 13 (r=1): ['주소', '서울시노원구동일로228길35']
  줄 14 (r=1): ['청수빌딩 204호']               ← 연속 ✓

PII 탐지 결과:
  전화번호: [010.4201.7478]  × 2
  주소: [서울시 노원구 동일로 228길  청수빌딩 204호]  (Zone 0, 스탬프)
  주소: [서울시노원구동일로228길35  청수빌딩 204호]   (Zone 1, 송장) ← 청수빌딩 204호 포함 ✓
```

## 4. 기술 상세

### `_split_box_by_words()` 적용 시점 변경 이유

- 원본 박스는 공간적 위치 정보가 OCR_Setting.py와 동일 → 클러스터링 결과도 동일해짐
- PII 탐지는 원본 텍스트 기반 full_text에서 수행 → 패턴 매칭 정확도 향상
- 마스킹은 PII 박스만 단어별 분리 → 불필요한 배경 훼손 최소화

### 패턴 14번 (주소 보강) 동작 확인

```
"서울시노원구동일로228길35  청수빌딩 204호" 에서:
- [가-힣]{2,3}시 → "서울시" ✓
- (?:[가-힣A-Za-z0-9]{1,6}[\s,]*){1,2} → "노원구" (1회) ✓
- [가-힣0-9]{1,10}로 → "동일로" ✓
- [\s,]*\d+ → "228" ✓
- (?:길[\s,]*\d+)? → "길35" ✓
- (?:[\s,]*[가-힣A-Za-z0-9]+(?:빌딩|...))? → "  청수빌딩" ✓ (공백 포함)
- (?:[\s,]*(?:\d+층|\d+-?\d*호|...))? → " 204호" ✓
```

| `ocr_mask/pipeline_detail_view.py` | `_mux_audio()` — `'ffmpeg'` 하드코딩 → `_get_ffmpeg_bin()` 사용 |

### 2-6. 영상 재생 불가 2차 원인 수정 (이번 세션 추가)

**실제 근본 원인 (2차 분석):**

`_mux_audio()`에서 `subprocess.run(cmd, capture_output=True, text=True)` 실행 시:
- ffmpeg 자체는 정상 실행 (returncode=0, H.264 파일 생성)
- 하지만 ffmpeg stderr에 한글 파일명(UTF-8)이 포함됨
- `text=True` 기본 인코딩 = Windows cp949 → **UnicodeDecodeError 발생**
- `except FileNotFoundError:` 로 잡히지 않아 예외 전파
- `_render_overlay_video()` → `run_detail_view()` → `local_worker.py` except로 흡수
- 결과: ffmpeg 성공했어도 H.264 파일이 최종 위치에 저장되지 않음 → 기존 mp4v 파일 유지

**수정:**

| 파일 | 변경 |
|------|------|
| `ocr_mask/pipeline_detail_view.py` | `subprocess.run()`에 `encoding='utf-8', errors='replace'` 추가 → cp949 디코딩 오류 방지 |
| `ocr_mask/pipeline_detail_view.py` | `except Exception as e:` 추가 → FileNotFoundError 외 모든 예외도 fallback 처리 |
| `output_file/563ea2ac-..._상세보기.mp4` | mp4v → H.264(avc1) + AAC 오디오 직접 재인코딩 완료 |

### _apply_mask_to_image 핵심 로직

```python
def _apply_mask_to_image(src_path, detections, output_path):
    img = cv2.imread(src_path)
    for det in detections:
        # polygon 우선 (회전 텍스트 박스 정밀 처리)
        poly_raw = det.get("polygon_json")
        if poly_raw:
            polygons = json.loads(poly_raw)
            for poly in polygons:
                pts = np.array(poly, dtype=np.int32).reshape((-1, 1, 2))
                cv2.fillPoly(img, [pts], (0, 0, 0))
        else:
            # bbox fallback (bbox_x + bbox_w → x2)
            x1, y1 = int(det["bbox_x"]), int(det["bbox_y"])
            x2, y2 = int(x1 + det["bbox_w"]), int(y1 + det["bbox_h"])
            cv2.rectangle(img, (x1, y1), (x2, y2), (0, 0, 0), -1)
    cv2.imwrite(output_path, img)
```

## 2-13. 미리보기 실패 오류 및 워터마크 규칙 수정 (260616 세션 추가)

### 에러 증상
프론트엔드에서 특정 개인정보에 대한 미리보기를 클릭했을 때, `미리보기 생성 실패 — 원본 파일 없음 또는 선택 PII 없음` 이라는 에러와 함께 `mask_preview` Job이 실패(failed)하는 문제 발생.

### 원인
1. 사용자가 미리보기 버튼을 눌렀을 때, 프론트엔드가 DB에 해당 PII의 `is_user_selected=True`를 저장하지 않은 상태로 `mask_preview` API를 호출할 수 있음.
2. 백엔드는 DB에 저장된 `is_user_selected=True`인 항목만 골라서 Colab 워커에 전달하므로, Colab은 '선택된 PII가 없다'고 판단하여 `run_preview()`에서 `None`을 반환.
3. 이로 인해 Colab 워커에서 `ValueError`를 발생시키고 실패로 처리됨.
4. 추가로, '상세페이지의 본 처리(마스킹) 시에는 안 보이는(비가시) 워터마크만 적용되어야 한다'는 요구사항과 달리, `WATERMARK_VISIBLE = True`로 되어 있어 가시 워터마크도 함께 적용되는 상태였음.

### 해결 및 변경 내용

**1. `backend/controllers/analysis.py` 및 `backend/services/analysis.py`**
- `POST /analysis/jobs/{job_id}/mask-preview` API가 선택적(optional)으로 JSON 본문(`{"pii_id": "..."}`)을 받을 수 있도록 수정.
- 전달된 `pii_id`가 있을 경우, 이를 `analysis_jobs` 테이블의 `message` 컬럼에 `target_pii_id:{pii_id}` 형태로 임시 기록하여 워커 큐에 넘김. (DB의 사용자 선택 `is_user_selected` 상태를 훼손하지 않음)

**2. `backend/services/worker.py`**
- `get_mask_job_context()` 함수에서 `message` 컬럼을 파싱하여 `target_pii_id:`가 존재하면, DB의 선택 여부와 무관하게 해당 `pii_id` 단일 항목만 `selected_pii_ids`로 강제 지정하여 Colab에 전달.
- 이를 통해 프론트엔드가 특정 `pii_id`만 미리보기 요청을 하면 해당 구간(앞뒤 3초, 총 6초)만 정상적으로 미리보기가 제공됨.

**3. `ocr_mask/colab_pipeline_mask.py` 워터마크 설정 변경**
- 미리보기 시에는 `preview=True` 플래그로 인해 'PREVIEW 미리보기' 패턴과 비가시 워터마크가 정상 적용되고 있었음.
- 본 처리 시에는 가시 워터마크를 제거하고 비가시 워터마크만 적용하도록 파일 상단의 설정을 `WATERMARK_VISIBLE = False`로 변경.
